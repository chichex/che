arquitectura

Backend de ejecución

Subprocess + headless con stream-json. No PTY como base. che invoca cada CLI con su flag de modo no-interactivo, parsea el output estructurado, y normaliza eventos al modelo común del adapter.

PTY queda como escape hatch para casos específicos donde algún CLI degrade features en headless (slash commands que solo viven en TUI, etc.). No es default.

Validado con smoke test del 2026-05-06. Los 4 CLIs cargan skills, custom commands y project markdown del usuario al ejecutarse en headless. La transparencia que se buscaba (heredar config sin reconfigurar) no requiere PTY.
stack

CLIs soportados (v1)

CLIModo headlessCustom config detectada
claude claude -p --dangerously-skip-permissions ~/.claude/skills/<name>/SKILL.md
codex codex exec --skip-git-repo-check ~/.codex/skills/<name>/SKILL.md
gemini gemini -p "..." --yolo ~/.gemini/commands/<name>.toml + GEMINI.md en cwd
opencode opencode run "..." ~/.config/opencode/skills/<name>/SKILL.md

Convergencia: claude / codex / opencode usan el mismo formato SKILL.md (frontmatter YAML + markdown). Gemini va aparte (TOML para commands, markdown para context). El adapter por CLI puede compartir gran parte del código entre los tres primeros.

Caveats encontrados

  • codex exec requiere --skip-git-repo-check si el cwd no es un git repo trusted.
  • claude puede vivir en ~/.local/bin/, no asumir /opt/homebrew/bin/. El adapter detecta paths en runtime.
  • gemini 0.38.2 ya tiene custom commands funcionando en headless (PR #8305 mergeado). Antes era un riesgo, ya no.
modelo

Pipeline y step

Un pipeline es una secuencia ordenada de steps que va de una premisa A a un entregable B. Un step ejecuta un prompt y opcionalmente lo valida en loop.

Anatomía de un step

CampoValoresDefault
namestringrequerido
cliclaude · codex · gemini · opencoderequerido
kindprompt (texto libre) · skill (skill cargada del CLI)requerido
contentsi kind=prompt: el texto. si kind=skill: el nombre.requerido
inputtext · pr · issue · file · url · none · previous_output (solo desde step 2)requerido
validatorbloque opcional (ver abajo)none
max_loopsint — cuántas veces se reintenta hasta que el validator apruebe1

Validator (opcional)

Mismo shape que un step. El validator puede usar un CLI distinto al step principal — esto habilita cross-review real (claude escribe, codex revisa).

CampoValoresDefault
kindprompt · skillrequerido
contentstring · nombre de skillrequerido
cliuno de los 4, independiente del CLI del steprequerido
on_max_loopsfail · continue · pausefail

Cuando hay validator, el step entra en loop:

  1. Ejecuta el step principal con la premisa + outputs anteriores.
  2. Pasa el output al validator.
  3. Si el validator aprueba → step termina exitoso, sigue el siguiente.
  4. Si el validator rechaza → re-ejecuta el step principal con prompt original + feedback del validator como contexto adicional, hasta max_loops veces.
  5. Si se agota max_loops sin aprobación → aplica on_max_loops:
    • fail: corta el pipeline, marca el step como fallido.
    • continue: avanza con la última versión, marca el step como "no convergido".
    • pause: para el pipeline y pide intervención humana.
decisiones cerradas

Lo que ya está cerrado (2026-05-06)

  • Backend: subprocess + headless con stream-json. PTY descartado como base.
  • CLIs: 4 backends — claude, codex, gemini, opencode.
  • Step: elige cli + kind (prompt o skill) + content + input.
  • Input del step: uno de text / pr / issue / file / url / none. Desde el step 2 se suma previous_output (solo el inmediato anterior). Sin "premisa" abstracta a nivel pipeline — cada step declara qué necesita.
  • Validator: opcional, con CLI independiente del step (cross-review habilitado).
  • on_max_loops: configurable por step, default fail.
  • Re-loop: el step se reinvoca con prompt original + feedback del validator.
  • Gates humanos: no soportados en v1. El humano interviene solo al disparar el pipeline (proveyendo el input que el step 1 pide) y al final (revisar entregable).
  • Formato del pipeline: YAML estricto como artefacto canónico. Parser que rechaza type coercion y campos desconocidos. La decisión no es irreversible — internamente che parsea a una struct, exportar a JSON o TOML se puede sumar después si aparece la demanda.
para más adelante

Preguntas que no bloquean el wizard

  • Pasaje de output entre steps: ¿el output del step N entra como input del step N+1 automáticamente, o cada step se invoca con la premisa original + un selector explícito tipo {{ steps.N.output }}?
  • Branching / condicionales: ¿pipelines siempre lineales en v1?
  • Skill inexistente referenciada: ¿validación al guardar el pipeline o al runtime?
  • Persistencia de outputs intermedios: ¿en memoria, en disco, ambos? ¿para audit trail, retry, debugging?
  • Definición del pipeline: ¿YAML, TOML, JSON, UI form-based, todas?
  • Concurrencia: ¿pueden correr varios pipelines en paralelo? ¿steps en paralelo dentro de un pipeline?

Reglas del wizard

  • Skill picker filtrado por CLI. Cuando el usuario elige cli: claude, el picker lista solo ~/.claude/skills/*. Cuando elige cli: gemini, lista los ~/.gemini/commands/*.toml. El campo se llama "skill" en la UI por uniformidad.
  • Default max_loops = 1. Si no hay validator, max_loops es 1 implícito. El campo solo aparece cuando hay validator activado.
  • Default validator CLI. El picker arranca con el mismo CLI que el step, pero sugiere visualmente "usar un CLI distinto" (cross-review). El usuario decide.
  • Validación en el wizard: si se referencia una skill que no existe en el CLI elegido, el wizard la marca en rojo y no deja avanzar.
  • Edición posterior: desde el resumen se puede ir a un step y editarlo, sin perder los demás.

YAML del pipeline

Las tres formas del wizard (TUI, web, edición directa) producen el mismo archivo. Vive en ~/.che/pipelines/<nombre>.yaml. 

name: triage-checkout-flow
description: |
  Toma una métrica anómala del flujo de checkout y propone
  causa probable.

steps:
  - name: collect-signals
    cli: claude
    kind: skill
    content: collect-signals
    input: text           # el step 1 pide texto al disparar
    validator:
      cli: codex
      kind: prompt
      content: |
        Revisá el output del step. ¿Las señales recopiladas son
        suficientes para hipotetizar una causa? Respondé OK si
        sí, o listá qué falta.
    max_loops: 3
    on_max_loops: fail

  - name: correlate-deploys
    cli: claude
    kind: skill
    content: correlate-deploys
    input: previous_output

  - name: hypothesize
    cli: claude
    kind: prompt
    content: |
      A partir del input recibido, proponé una causa probable
      de la caída del approval rate. Justificá con evidencia
      concreta.
    input: previous_output
    validator:
      cli: codex
      kind: skill
      content: cross-review-hypothesis
    max_loops: 3
    on_max_loops: pause
