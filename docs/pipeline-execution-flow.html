Flujo de ejecución de pipelines

Diagrama de estados, transiciones, bifurcaciones y validaciones de todo lo que es ejecución de pipelines: recolectar el input del primer step, preflight de dependencias, correr cada step en subprocess con streaming, loops del validator, persistencia de artefactos por run, cancel, finalización y vuelta al menu. No cubre la creación ni edición del YAML — eso vive en manage-pipelines-flow.html.

Compañero de manage-pipelines-flow.html (que cubre el wizard S1–S4 y el lister de drafts/ready) — éste cubre lo que pasa cuando un pipeline ready se dispara desde My pipelines: enter sobre el row → este flow.

Punto de entrada único: My pipelines → enter sobre un row con chip ready. No hay invocación por subcomando ni por argumento — el run vive dentro de la TUI principal de che.

artefactos

Persistencia del run

Cada run produce un directorio dedicado bajo ~/.che/runs/<slug>/<run-id>/. La idea es que un crash, un kill abrupto o un terminal cerrado dejen suficiente material en disco como para auditar qué pasó sin depender de que la TUI siga viva: stdout/stderr se flushean en cada línea, el manifest se reescribe atómicamente en cada transición de step, y el resultado del step queda parseado en un archivo aparte para que el step siguiente lo lea sin tener que re-tokenizar el stdout.

Un run = un directorio inmutable. Una vez terminado (done / failed / cancelled), nada lo edita. Re-correr el pipeline genera un run-id nuevo. Esto convierte ~/.che/runs/<slug>/ en un historial natural ordenado por timestamp.

Layout en disco

~/.che/runs/triage-checkout-flow/
  2026-05-08T14-32-11/                     # run-id = ISO sin colons
    manifest.yaml                          # estado del run (atómico)
    input.yaml                             # input recolectado en R1
    step-01.stdout.log                     # raw stream del subprocess
    step-01.stderr.log
    step-01.events.jsonl                   # eventos parseados (stream-json)
    step-01.result.yaml                     # resultado consumible por step N+1
    step-01.validator.01.stdout.log         # validator loop iter 1
    step-01.validator.01.events.jsonl
    step-01.validator.01.verdict.yaml        # {verdict: ok|fail, feedback: ...}
    step-02.stdout.log
    ...
  2026-05-08T16-44-02/                     # otro run del mismo pipeline
    ...

Schema del manifest.yaml

Reescrito atómicamente (.tmp + os.Rename) en cada transición de step y al cerrar (done / failed / cancelled). La estructura es plana para que un cat rápido te diga el estado actual sin parsearlo entero. 

run_id: 2026-05-08T14-32-11
pipeline_slug: triage-checkout-flow
pipeline_path: ~/.che/pipelines/triage-checkout-flow.yaml
started_at: 2026-05-08T14:32:11-03:00
finished_at: 2026-05-08T14:38:42-03:00   # vacío mientras status=running
status: done                              # running | done | failed | cancelled | interrupted
current_step: 4                            # 1-indexed; congelado al terminar
total_steps: 4
che_version: 0.0.96
steps:
  - idx: 1
    name: collect-signals
    cli: claude
    status: done                          # pending | running | done | failed | cancelled
    started_at: 2026-05-08T14:32:11-03:00
    finished_at: 2026-05-08T14:34:03-03:00
    exit_code: 0
    validator: null                       # bloque presente solo si el step lo tiene
  - idx: 2
    name: cluster-anomalies
    cli: claude
    status: done
    started_at: 2026-05-08T14:34:04-03:00
    finished_at: 2026-05-08T14:36:21-03:00
    exit_code: 0
    validator:
      cli: gemini
      loops_run: 2
      max_loops: 3
      on_max_loops: fail
      final_verdict: ok                  # ok | fail | timeout

Schema del step-NN.result.yaml

El "output canónico" del step. Lo escribe el runner al cerrar el subprocess, parseando los eventos finales de events.jsonl (assistant message final + cualquier verdict bloque YAML que el step haya emitido). El step N+1 con input: previous_output lee este archivo, no el stdout crudo — así no nos importa el verbose ni los tool-use chunks. 

step_idx: 1
cli: claude
kind: skill
content: collect-signals
duration_ms: 112430
exit_code: 0
output: |
  47 anomalías encontradas en logs de checkout entre 14:00 y 14:32...
  Cluster A (latencia P99 > 800ms): ...
  Cluster B (timeouts gateway): ...
verdict: null                              # solo presente si fue salida de un validator

Schema del input.yaml (R1)

step_idx: 1
step_input_kind: issue                       # text|pr|issue|file|url|none
value: "chichex/che#42"                     # lo que el usuario tipeó / eligió
resolved_at: 2026-05-08T14:32:09-03:00
resolved_payload: |                            # el contenido fetched (issue body, file contents, etc.)
  Title: "checkout 500s en europa"
  Body: ...

Cuándo se escribe cada cosa

EventoArchivos tocados
Entrar a R3 (start del run) Crear dir, escribir manifest.yaml con status: running y steps[].status: pending; copiar input.yaml resuelto.
Step N arranca Update steps[N-1].status: running, started_at. Manifest atómico.
Línea de stdout/stderr del step Append a step-NN.stdout.log / .stderr.log. Sin atómico, sin batching: file.WriteString(line) + Sync() por línea.
Evento parseado del stream Append a step-NN.events.jsonl. Sirve para reconstruir la UI o auditar tool-use sin volver a parsear stdout.
Step N termina (exit) Escribir step-NN.result.yaml; update manifest steps[N-1].status: done|failed, exit_code, finished_at.
Validator loop K arranca Update manifest validator.loops_run: K; subprocess separado; eventos a step-NN.validator.0K.*.
Validator loop K cierra Escribir step-NN.validator.0K.verdict.yaml; update manifest. Si verdict: ok o K == max_loops, salir del loop.
Run termina (cualquier salida) Update manifest status + finished_at. Cierre de file handles. Trigger de GC.

Garantías y no-garantías

  • Atomicidad del manifest: .tmp + rename. Un crash entre dos transiciones deja un manifest viejo pero coherente, nunca uno trunco.
  • Logs no son atómicos. En un kill -9 podés perder hasta lo que estaba en buffer del runtime de Go (Sync() reduce a casi cero pero no garantiza). Asumido — el manifest es la fuente de verdad.
  • No hay resume v1. Un run con status: running al boot es un interrupted. La siguiente vez que el lister lea el manifest reescribe el status a interrupted y deja el dir como está; el usuario decide borrarlo o no.
  • GC: al iniciar un run nuevo, mantener los últimos 10 dirs por pipeline (last_saved_at desc); más viejos → os.RemoveAll. Configurable con env CHE_RUN_HISTORY=N, default 10.
  • Concurrencia: dos runs simultáneos del mismo pipeline son posibles porque el run-id los separa físicamente. No hay lock — el segundo no espera al primero. Out of scope v1.
lifecycle

Ciclo de vida end-to-end

Lo que pasa entre el enter sobre un row ready y la vuelta al menu, mirado como una secuencia de bloques en vez de un grafo. Los transitions ya están en el state diagram; esta sección documenta qué hace el código en cada bloque.

FaseTriggerOutput
1 · Resolución enter sobre row ready en My pipelines. Re-parsear ~/.che/pipelines/<slug>.yaml (no usar la copia en RAM del lister — puede estar stale si el usuario abrió $EDITOR). Validar IsValid() de nuevo. Si falla → toast en el lister, no se entra a R1.
2 · Input (R1) Pipeline válido + step 1 con input != none. Pantalla R1 que pide el input según el kind (textarea, picker de PR/issue via gh, file picker via TUI, URL input). Si input: none, R1 se skipea y va directo a R2.
3 · Resolve Usuario confirma input. Resolver el payload: gh issue view <ref> --json title,body, os.ReadFile, http.Get. Falla de fetch → R1 con error inline. Éxito → guardar a input.yaml in-memory (todavía no hay run-id).
4 · Preflight (R2) Input resuelto. Recorrer todos los steps y validar:
· CLI installed (reusar internal/skills.Detect).
· Si kind: skill, la skill existe en ese CLI.
· Si algún step requiere gh y gh auth status falla, marcar.
R2 muestra una checklist (✓/✗ por chequeo); si todos pasan, "siguiente"; si alguno falla, opciones volver a My pipelines / correr che doctor.
5 · Init run Preflight OK. Generar run-id (time.Now().UTC().Format("2006-01-02T15-04-05")); crear ~/.che/runs/<slug>/<run-id>/; escribir input.yaml + manifest.yaml inicial; trigger GC (mantener últimos N).
6 · Step loop (R3) Manifest creado. Por cada step del pipeline (en orden): resolver input (kind del step 1 ya está; para idx ≥ 1 con previous_output leer step-(N-1).result.yaml); spawn subprocess; tee stdout → .stdout.log + parser stream-json; tee stderr → .stderr.log; al exit, escribir result.yaml + actualizar manifest. Si tiene validator, entrar al validator loop.
7 · Validator loop Step terminó con exit_code: 0 + tiene validator. Spawn validator con step-NN.result.yaml/output como input. Parsear verdict: ok|fail + feedback al final del stdout. Si ok → siguiente step. Si fail y loops_run < max_loops → re-correr el step pasando el feedback como contexto extra. Si hit max_loops → aplicar on_max_loops.
8 · Terminal All steps done · o un step/validator falló · o el usuario hizo ctrl+c. Cerrar manifest con status: done|failed|cancelled, finished_at; transición a R4 (RunDone), RF (Failed) o RC (Cancelled). En todos los casos, el pipeline-yaml en ~/.che/pipelines/ queda intacto.

R1

InputPrompt — recolección del input del step 1

screen 1 / 4 del runner
Trigger
R0 emite "run pipeline X" + step[0].input != "none".
Render dinámico
  • text · textarea multiline (mismo estilo que S1 description). Hint: "el step recibirá este texto en stdin / como prompt".
  • pr · single-line input que pide owner/repo#NNN. Validación: existe vía gh pr view.
  • issue · análogo a pr, gh issue view.
  • file · file picker (lista $PWD + ~/ con tab para navegar). Validación: existe + es regular file + size < 10 MiB (configurable env CHE_MAX_INPUT_SIZE).
  • url · single-line. Validación de schema (http/https). Fetch eager con timeout 10s al confirmar.
  • none · screen no aparece — skip a R2.
Validaciones
  • Input no vacío (excepto none).
  • Resolución eager: si gh / http / os.ReadFile falla, mostrar el error inline y quedarse en R1.
  • Resolved payload no es vacío post-fetch (un PR sin body sigue siendo válido — solo bloqueamos si err != nil).
Transiciones
  • ctrl+s / enter sobre el último foco · validar + resolver + ir a R2.
  • esc · vuelve a R0 (sin run iniciado, sin manifest creado).
  • ctrl+c · salida total de che.
Estado del modelo
Tras transición exitosa: RunModel.Input seteado con {Kind, Value, ResolvedPayload}. Sin escribir disco todavía.
R2

Preflight — chequeos de dependencias

screen 2 / 4 del runner
Trigger
R1 confirmó el input · o R1 se skipeó por input: none.
Render
Lista de chequeos con ✓ / ✗ / ⏳ (mientras se evalúan). Cada row tiene una línea adicional con el remedio si falla.
Preflight de triage-checkout-flow ✓ claude · v1.4.2 ✓ gemini · v0.8.1 (validator del step 2) ✗ codex · no instalado remedio: brew install codex · o cambiá el step 3 a otro CLI ✓ skill collect-signals en claude ✓ skill cluster-anomalies en claude ✗ skill summarize-checkout en codex remedio: edit pipeline (cambiar a prompt) o instalar la skill ✗ gh · auth status fallido remedio: gh auth login 3 problemas · esc volver · r reintentar · d abrir doctor
Chequeos
  • cli installed por cada CLI distinto en steps[].cli y steps[].validator.cli.
  • skill exists por cada step/validator con kind: skill.
  • gh auth si algún step tiene input: pr|issue.
  • file readable si input.kind == "file" (re-check post R1, defensivo: el archivo pudo desaparecer entre R1 y R2).
  • disk space ≥ 100 MB libres en ~/.che/runs. Si no, warning amarillo (no bloqueante).
Transiciones
  • Todos verdes → enter avanza a R3 (start del run).
  • Algún rojo → enter bloqueado. r reintenta los chequeos. d sale a che doctor (suspende TUI). esc vuelve a R0.
  • Solo warnings amarillos → enter avanza con confirm modal "continuar igual".
R3

Running — full-screen takeover con steps + log streaming

screen 3 / 4 · loopable por validator
Trigger
R2 OK → init run dir → spawn step 1.
Layout
┌ Run: triage-checkout-flow step 2/4 ┐ │ ▸ collect-signals ✓ done 2.3s │ │ ▸ cluster-anomalies ⏳ running loop 1/3 │ │ review ✕ pending │ │ summarize ✕ pending │ ├───────────────────────────────────────────────────────┤ │ [2/4] cluster-anomalies (claude / skill) │ │ > Reading collect-signals.json... │ │ > Found 47 anomalies clustered into 6 groups │ │ > Validator (gemini) loop 1/3... │ │ █ │ ├───────────────────────────────────────────────────────┤ │ ctrl+c cancel · ctrl+l clear · ↑↓ scroll │ └───────────────────────────────────────────────────────┘
  • Header: nombre del pipeline, contador step N/M.
  • Steps tracker: rows compactos por step con status icon + nombre + duración + (si validator) loop K/max.
  • Log pane: viewport scrolleable con las últimas N líneas del subprocess actual (parseadas desde stream-json cuando aplica). Auto-scroll al fondo a menos que el usuario haya scrolleado arriba (sticky-bottom).
  • Footer: hints de keys.
Render del log pane
  • Para CLIs con stream-json (claude/codex): parser tolerante (ver memory bufio.Scanner 64 KiB: buffer 1 MiB) emite líneas humano-legibles (> assistant text, · tool: read_file).
  • Para CLIs sin stream-json (gemini text mode): mostrar el stdout crudo, una línea por línea.
  • Stderr en rojo dimmed, intercalado con stdout.
  • Reusar internal/output como sink — stdout=payload, stderr=logs (ver memory logger unificado).
Transiciones
  • ↑/↓ · scroll del log pane (desactiva sticky-bottom).
  • g · vuelve sticky-bottom y salta al fondo.
  • ctrl+l · limpia el viewport (los logs en disco quedan; sólo afecta render).
  • tab · cycle entre steps tracker y log pane (foco visual; afecta de qué step se renderean los logs).
  • ctrl+c · abre RC modal (cancel confirm).
  • Auto: step exitoso + sin validator → siguiente step. Step exitoso + validator → validator loop. Step exit_code != 0 → RF.
Modelo
RunModel.Steps[i] con Status, StartedAt, FinishedAt, ExitCode, ValidatorState; RunModel.LogBuffer[i] ring buffer de las últimas 2000 líneas por step (memoria; el archivo en disco es full).

RF

Failed — terminal de error

terminal
Trigger
Algún step terminó con exit_code != 0, o un validator devolvió fail y on_max_loops: fail, o R2 detectó un problema fatal y el usuario eligió no continuar.
Render
Resumen rojo con el mismo layout que R4 pero marcando el step que falló y mostrando las últimas 20 líneas de su stderr.log + stdout.

Transiciones
  • enter / esc · vuelve a R0.
  • l · $PAGER sobre el stderr.log del step que falló.

RC

Cancelled — confirm de cancelación

modal sobre R3
Trigger
ctrl+c en R3 (siempre — el run no es "salida accidental").
Render
Modal con 3 opciones:
  • abort & save — SIGTERM al subprocess, gracia 5s, SIGKILL si no salió. Manifest se cierra con status: cancelled + el step en curso queda con status: cancelled. Ir a un screen tipo RF pero con tono amarillo.
  • back to run — cierra el modal, el run sigue.
  • exit che — abort & save + salida total del proceso (no vuelve al menu).
Comportamiento
  • El runner SIEMPRE intenta SIGTERM antes que SIGKILL — los CLIs pueden estar haciendo cleanup (cerrar archivos, terminar tool calls, llamar abort de la API).
  • Gracia configurable env CHE_KILL_GRACE, default 5s.
  • Logs flusheados antes del exit.
RP

Paused — validator agotó max_loops con on_max_loops=pause

modal sobre R3 · sólo si el step lo declara
Trigger
Step con validator + on_max_loops: pause agotó el contador de loops.
Render
Modal con el último verdict.feedback del validator + 3 opciones:
  • continuar — aceptar el último output del step, pasar al siguiente. Manifest registra final_verdict: human-override.
  • retry — reset del contador y otro intento (consume max_loops nuevos). Útil si el usuario quiere editar el prompt antes (out of scope v1: editar el prompt mid-run).
  • abort — equivalente a on_max_loops: fail. Va a RF.
Estado del modelo
El subprocess ya no está vivo cuando el modal aparece — los outputs ya están escritos. RP es 100% decisión humana.
bifurcaciones

Decisiones que cambian la UI o el comportamiento

B1 — Tipo de input del step 1 cambia R1

input: text Textarea multiline. Resolución: el texto crudo es el payload.
input: pr / issue Single-line input + helper que sugiere repos del $PWD (lee git remote -v + cache de gh repo list). Resolución: gh pr view --json title,body,comments. Falla → R1 con error inline.
input: file File picker minimalista con tab para nav + filtro inline. Resolución: os.ReadFile + check de tamaño.
input: url Single-line. Resolución: http.Get con timeout 10s, body como string. Status no-2xx → error inline.
input: none R1 se skipea. input.yaml tiene step_input_kind: none + resolved_payload: "".

B2 — Validator on/off cambia el flow del step

validator: nil Step termina → siguiente step. manifest.steps[i].validator ausente.
validator: presente Step termina con exit_code: 0 → spawn validator con el output del step como input. Parsear verdict. Loopear hasta ok o max_loops. Si fail final → aplicar on_max_loops.

B3 — Resolución de previous_output

step idx == 0 Imposible por construcción — IsValid ya bloqueó esto en el wizard. Defensive: si llegara a aparecer, RF inmediato con error "previous_output en step 0".
step idx ≥ 1 Leer step-(N-1).result.yaml/output y pasarlo como input del step N. Si el step previo terminó con validator activo, leer del último loop exitoso.

B4 — on_max_loops

fail Manifest final_verdict: fail · run termina en RF.
continue Manifest final_verdict: fail-but-continued · acepta el último output del step y pasa al siguiente. Útil para validators "mejor esfuerzo" donde la última iteración igual sirve.
pause Trigger de RP modal · espera decisión humana.

B5 — Render del log pane según CLI

claude / codex (stream-json --verbose) Parser de eventos: {type:assistant, content} → línea humana, {type:tool_use, name, input} → "· tool: name". Buffer 1 MiB del scanner para evitar el truncate silencioso de 64 KiB (ver memory).
gemini / opencode (text) Stdout crudo line by line. Sin parsing.
modelo

Modelo interno

Una struct vive en el bubbletea program del runner. Los handles a los subprocesos y los file abiertos viven adentro — al cerrar el program (cualquier salida) se hace flush + close de todo. 

// internal/runner/model.go
type RunModel struct {
    Screen   RunScreen           // R1 | R2 | R3 | R4 | RF | RC | RP
    Pipeline wizard.Pipeline     // reusa el struct del wizard

    // R1
    Input    InputState          // kind, value, resolvedPayload

    // R2
    Preflight []PreflightCheck   // {Label, Status, Remedy}

    // R3
    RunID      string            // 2026-05-08T14-32-11
    RunDir     string            // ~/.che/runs/<slug>/<id>/
    Steps      []StepRun          // status, started, finished, exit, validator
    Active     int               // idx del step en curso (0 = step 1)
    LogBuffers []*RingBuffer      // 2000 líneas por step, en RAM

    // log pane focus
    LogFocus      int            // idx del step cuyos logs se renderean
    StickyBottom  bool           // auto-scroll al fondo

    // modales
    CancelModal   bool
    PauseModal    *PauseState     // nil si no hay pause activa

    // process handles (cerrados al exit del program)
    activeCmd     *exec.Cmd
    stdoutFile    *os.File
    stderrFile    *os.File
    eventsFile    *os.File
}

type StepRun struct {
    Idx        int               // 1-indexed para manifest
    Status     StepStatus        // pending|running|done|failed|cancelled
    StartedAt  time.Time
    FinishedAt time.Time
    ExitCode   int
    Validator  *ValidatorRun     // nil si el step no tiene
}

type ValidatorRun struct {
    LoopsRun     int
    MaxLoops     int
    OnMaxLoops   string          // fail|continue|pause
    FinalVerdict string          // "" mientras no termine; ok|fail|timeout|human-override
    LastFeedback string          // del último verdict.yaml
}
paths

Naming, paths y conflictos

AspectoDecisiónNotas
run-id time.Now().UTC().Format("2006-01-02T15-04-05") Sortable, filename-safe (sin colons), sin colision en granularidad de 1s. Si dos runs caen en el mismo segundo (edge case), el segundo agrega sufijo -2.
Run dir ~/.che/runs/<slug>/<run-id>/ Permisos 0700 en dirs, 0600 en files. Crear lazy.
Atomicidad del manifest manifest.yaml.tmp + os.Rename Mismo patrón que el wizard. Tests reusables.
Logs Append continuo + Sync() por línea No atómicos. Trade-off conocido — losses bajo SIGKILL son aceptables.
GC Mantener últimos N runs por pipeline (default 10, env CHE_RUN_HISTORY) Trigger al iniciar un run nuevo. os.RemoveAll de los más viejos.
Concurrencia Sin lock. Dos runs simultáneos posibles. Out of scope v1. Cada uno con su run-id, no se pisan.
subprocess

Cómo se invoca cada CLI

Convenciones por CLI. Todas pasan el input/payload por stdin para evitar problemas de escaping de shell. La elección entre stream-json y text sigue la regla de la memory Invocación al CLI de claude: stream-json para ejecuciones largas con tool use; text si el step explícitamente lo pide (no expuesto v1, default stream-json donde aplica).

CLIComandoModo de output
claudeclaude -p <skill-or-prompt> --output-format stream-json --verboseevents.jsonl + stdout en log
codexcodex exec --json <skill-or-prompt>events.jsonl + stdout en log
geminigemini -p <prompt> · si kind: skill → gemini /<skill> (alias de commands TOML)stdout crudo (text mode)
opencodeopencode run <prompt>stdout crudo

Stdin del subprocess: el resolved_payload del input (R1) o el output del previous_output step. Para CLIs que no leen stdin, se concatena al final del prompt con un separador (\n\n---\n\n).

Validator: mismo patrón, pero el prompt incluye un preámbulo que pide explícitamente el bloque YAML verdict al final. El parser de verdict.yaml es tolerante: busca el último bloque YAML válido en el output que tenga la clave verdict; si no encuentra ninguno, asume fail con feedback "no verdict block".

H4

R3 spawn básico — un step, sin streaming, exit code

deps: H3
Como usuario, quiero que el runner ejecute el primer step en subprocess y muestre su exit, para tener un loop completo end-to-end aunque sea de un solo step y sin streaming.

Notas de implementación

  • Modo blocking: cmd.Run(), no cmd.Start(). Stdout/stderr se capturan completos a archivo + buffer en RAM.

  • Exit ≠ 0 → transición a RF (placeholder con error).

H5

Como usuario, quiero ver el output del step llegando línea por línea en el log pane, para saber que el run está vivo y entender qué está haciendo el agente.

H7

Validator loop — max_loops + on_max_loops + RP modal

deps: H6
Como usuario, quiero que un step con validator loopee hasta que el verdict sea ok o se agoten los max_loops, para que el cross-review del wizard se haga efectivo en runtime.

Criterios de aceptación

H8

Manifest atómico + GC + recovery de runs interrumpidos

deps: H7
Como dev / ops, quiero que el manifest sobreviva crashes coherente, que los runs viejos no se acumulen, y que un kill abrupto deje un estado leíble, para poder confiar en el historial de runs.

Criterios de aceptación

  • Cada update del manifest pasa por writeManifestAtomic: .tmp + os.Rename.
  • Updates triggeados en: start del run, start/end de cada step, start/end de cada validator loop, end del run.
  • GC: al iniciar un run nuevo, escanear ~/.che/runs/<slug>/, mantener los últimos N (default 10, env CHE_RUN_HISTORY) por mtime, os.RemoveAll el resto.
  • Recovery: al boot del lister (R0), por cada manifest con status: running y started_at > 1h, reescribir a interrupted. Defensive: si started_at < 1h, dejar como está (puede ser otro proceso de che corriendo en paralelo).
  • Tests cubriendo: crash entre dos updates → manifest queda en estado anterior consistente, no trunco.

Smoke manual

  1. Run, interrumpir con kill -9 al PID de che mid-step.
  2. cat ~/.che/runs/<slug>/<id>/manifest.yaml → status=running.
  3. Re-abrir che → My pipelines → ver row con chip "interrupted" (manifest se reescribió).
  4. Disparar 12 runs en serie → ver que solo quedan los últimos 10.

Test e2e

  • Disparar run, simular SIGKILL (os.Process.Kill sobre el PID de che en un test integrationcito); reload del lister; assert: manifest status=interrupted.
  • GC: pre-armar 15 dirs con mtimes escalonados, disparar run; assert: solo los últimos 10 quedan + el nuevo.
  • Atomicidad: kill entre writeFile(.tmp) y Rename (vía os.Setenv("CHE_FAULT_INJECT_BEFORE_RENAME", "1")); assert: manifest viejo intacto, .tmp huérfano (cleanup en boot).

Out of scope

Cleanup de .tmp huérfanos automático (post-v1, no crítico). Compresión de logs viejos.

H9

Polish del cancel — graceful shutdown + cleanup de handles

deps: H8
Como usuario, quiero que un cancel mid-run no deje archivos truncos, procesos zombie, ni la TUI rota, para poder cancelar sin miedo.

Criterios de aceptación

  • RC modal de H4 se promueve a 3 opciones bien definidas: abort & save · back to run · exit che.
  • Abort & save: SIGTERM, gracia CHE_KILL_GRACE (default 5s), SIGKILL si no salió. Flush + close de stdout/stderr/events files. Manifest cierra status: cancelled, step en curso status: cancelled, exit_code: -1.
  • Exit che: lo de arriba + os.Exit(0).
  • Back to run: cierra el modal, run sigue.
  • Si el subprocess salió por sí mismo entre el ctrl+c y la decisión del modal (race), tratar como step normal terminado.
  • Tras cancel, transición a un screen tipo RF pero con tono amarillo (RC-done): título "Run cancelado", lista de steps con el que se canceló marcado, opciones r retry / l log / esc volver.

Smoke manual

  1. Run con fake CLI sleep 30; ctrl+c → modal → "back to run" → ver que el run sigue.
  2. Mismo, ctrl+c → "abort & save" → ver pantalla amarilla "Run cancelado".
  3. Mismo CLI pero que ignora SIGTERM (trap '' TERM); abort & save; verificar que se SIGKILL tras 5s.
  4. Después del cancel, ps aux | grep claude no debería tener huérfanos.

Test e2e

  • Fake CLI que ignora SIGTERM. Assert: tras abort, exit code refleja SIGKILL, manifest cancelled, todos los handles cerrados (no EBADF en re-abrir el archivo).
  • Race test: fake CLI sale exit 0 right antes del SIGTERM. Assert: manifest cierra done, no cancelled.
