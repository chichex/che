Flujo de gestión de pipelines

Diagrama de estados, transiciones, bifurcaciones y validaciones de todo lo que es gestión de pipelines: crear, persistir incremental, listar, reanudar drafts, editar pipelines ready, borrar. No cubre la ejecución (subprocess, stream-json, validators en loop, capture de output) — eso vive en su propio doc cuando llegue su turno.

Compañero de design.html (que cubre los mockups del wizard) — éste cubre la navegación: cómo se entra, cómo se sale, qué pasa en cada bifurcación, qué se persiste en cada paso.

Puntos de entrada del menu principal: 1. My pipelines (listar + accionar) · 2. Create pipeline (wizard de creación nueva). La etiqueta original See pipelines se reemplaza por Create pipeline; el placeholder de ítem 1 (My pipelines) se reemplaza por My pipelines.

autosave

Persistencia incremental

El wizard persiste el pipeline a disco a medida que el usuario lo llena, así una salida abrupta (terminal cerrado, ssh disconnect, ctrl+c sin confirm, crash, kernel panic) no pierde progreso. El archivo es el mismo YAML que un pipeline final, con la única diferencia de un bloque status opcional que indica dónde quedó el wizard. Cuando el pipeline está completo, el bloque status se omite y el archivo queda 100% limpio.

Un solo schema, un solo directorio. Mismo struct Pipeline, mismo path. La diferencia entre draft y final es la presencia (o ausencia) del bloque status.

Cuándo se guarda

  • Sincrónico — en cada transición de estado. S1→S2, S2→S2 al "+ agregar step", S2→S3, S3→S2 al editar, S2→S3 al guardar el step editado, S3 al borrar/reordenar step. Bloquea el render hasta confirmar el write.
  • Asincrónico — debounced 800ms en cada field change dentro de S1 y S2 (texto, toggle de validator, selección de skill, etc.). No bloquea input.
  • Nunca en cada keystroke crudo — eso lo cubre el debounce.

Dónde vive el archivo y desde cuándo

  • ~/.che/pipelines/<slug>.yaml · el mismo lugar y formato que un pipeline final.
  • Mientras el usuario no completó name en S1, el archivo no existe todavía — el wizard mantiene el modelo en RAM. La pérdida posible es lo tipeado en S1 (típicamente < 30s).
  • Apenas name tiene un valor derivable a slug, se crea el archivo en el path final. De ahí en adelante, todos los cambios persisten.
  • Permisos 0700 en el dir, 0600 en los archivos. Write atómico: escribir a <path>.tmp + os.Rename.

Cómo se determina draft vs ready — bloque status

Mientras el wizard tiene el pipeline abierto y todavía no se "Guardó pipeline" en S3, el archivo lleva un bloque status con la ubicación exacta dentro del wizard. Apenas el usuario confirma "guardar pipeline" (S3 → S4) y el pipeline pasa IsValid(), el bloque status se elimina del archivo. Esto significa:

  • Si el archivo tiene status → es un draft. My pipelines lo lista con chip draft. Al seleccionarlo, el wizard arranca en status.stage.
  • Si el archivo no tiene status → es ready. Chip ready. Al seleccionarlo, ejecuta.

No hay validación implícita por contenido para clasificar. El bloque status es la única señal — explícito y fiable, sin ambigüedad si una skill se borró o un CLI se desinstaló (esos casos siguen siendo ready pero ejecutable solo si las deps están).

Schema del bloque status

status:
  stage: step          # info | step | summary
  step_idx: 1           # solo si stage=step
  step_mode: create    # solo si stage=step — create | edit
  last_saved_at: 2026-05-07T14:32:11-03:00

name: triage-checkout-flow
description: Toma una métrica anómala...
steps:
  - name: collect-signals          # step completo
    cli: claude
    kind: skill
    content: collect-signals
    input: text
  - name:                            # step en construcción, faltan campos
    cli: claude
    kind: skill
    content: 
    input:

Pipeline final (sin status): 

name: triage-checkout-flow
description: ...
steps:
  - name: collect-signals
    ...

Concurrencia (dos wizards abiertos al mismo pipeline): out of scope v1. Last-write-wins de facto, sin lock, sin warning. Documentado.

Pérdida máxima posible

Con esta estrategia, en el peor caso (kill abrupto entre dos debounces con cambios in-flight), el usuario pierde ≤ 800ms de tipeo dentro de un campo. Toda transición de estado y todo cambio > 800ms tras el último cambio están persistidos. Si el crash sucede antes de que haya nombre (S1 sin completar), se pierde todo lo tipeado en S1 — pero eso es típicamente < 30s de progreso.

resume

Reanudar un pipeline incompleto — vía My pipelines

My pipelines (ítem 1 del menu) lista todos los archivos en ~/.che/pipelines/, distinguiendo visualmente cuáles están completos y cuáles incompletos. La acción al seleccionar depende del estado:

EstadoMarca visualAcción al seleccionar
Completo ready · sin marca extra Pide el input del step 1 → ejecuta el pipeline.
Incompleto draft · sub-label "last edited 12 min ago — paso 2 de 3" Abre el wizard en la screen donde más conviene continuar (ver "Screen target del resume" abajo).

Render del listado

Mezclados en una sola lista, ordenados por last_saved_at desc. Cada row muestra su chip de estado, el nombre, una línea con timestamp relativo ("12 min ago") y, si es draft, el stage donde quedó ("paso 2 de 3").

Screen target del resume

Cuando el usuario selecciona un draft, el wizard lee directamente el bloque status del YAML y arranca en la screen indicada por status.stage:

  • stage: info → S1 con name + description pre-cargados.
  • stage: step → S2 con step_idx y step_mode (create o edit) del bloque.
  • stage: summary → S3 con todo el pipeline pre-cargado.

No hay inferencia de "donde estaba" por contenido — el wizard lo escribió y el wizard lo lee. Si status está corrupto o apunta fuera de range, fallback a S1 con warning visible.

Acciones secundarias en My pipelines

  • enter sobre un draft · abre el wizard en la screen de status.stage.
  • enter sobre un ready · pide input del step 1 → ejecuta.
  • e sobre un ready · re-abre el wizard en mode=edit; el wizard re-introduce status: { stage: summary } al cargar. Al "guardar pipeline" en S3 lo vuelve a remover.
  • d sobre cualquiera + confirm · borra el archivo. Sirve para drafts huérfanos también.
  • y sobre cualquiera · abre el YAML en $EDITOR. Re-parsea al cerrar.

S2

StepEditor — definir un step

screen 2 / 3 · loopable
Modos
  • create: estás definiendo un step nuevo (último de la lista).
  • edit: viniste desde Summary a editar un step existente — al guardar se vuelve a Summary, no se loopa.
Campos
  • name · string · requerido
  • cli · enum claude|codex|gemini|opencode · requerido · sugerido solo entre los CLIs installed (reusar internal/skills.Detect)
  • kind · enum prompt|skill · requerido
  • content · string · requerido · si kind=skill es el nombre, si kind=prompt es el texto
  • input · enum dependiente de la posición del step (ver bifurcación B1)
  • validator · bloque opcional (ver bifurcación B2)
  • max_loops · int · solo visible si validator activo · default 3
  • on_max_loops · enum fail|continue|pause · solo visible si validator activo · default fail
Validaciones
  • name no vacío y único dentro del pipeline.
  • Si kind=skill: la skill debe existir en el CLI elegido (chequear con internal/skills) — si no, marcar el row en rojo y bloquear avance.
  • Si kind=prompt: content trimmed no vacío.
  • Si validator activo: todos los campos del validator validados con las mismas reglas.
  • max_loops entero ≥ 1.
Transiciones
  • + agregar step (ctrl+n): valida → push step a model.Pipeline.Steps → reabre S2 en blanco para step N+1. solo en mode=create
  • guardar y ver resumen (ctrl+s): valida → push si create / replace si edit → ir a S3.
  • atrás (esc en mode=create):
    • si es step 1 → vuelve a S1 descartando lo tipeado en este step.
    • si es step 2+ → vuelve al step previo en S2 mode=edit.
  • atrás (esc en mode=edit): vuelve a S3 sin guardar cambios del step.
S3

Summary — revisión y persistencia

screen 3 / 3
Trigger
Desde S2 con "guardar y ver resumen".
Render
  • Header: name + description + chip con el input que el step 1 va a pedir al disparar.
  • Lista de steps con: posición, nombre, CLI, kind+content, input, presencia de validator (con CLI + max_loops + on_max_loops si aplica).
  • Footer: path donde se va a guardar (~/.che/pipelines/<slug>.yaml).
Selección
↑/↓ mueve el cursor entre steps. El step apuntado es el target de "editar step" / "borrar step".
Transiciones
  • guardar pipeline (ctrl+s): correr IsValid(); si pasa → serializar sin status, escribir, ir a S4. Si falla → mostrar errores inline, quedarse en S3.
  • editar step (e sobre el step apuntado): ir a S2 en mode=edit con los campos del step pre-cargados.
  • + agregar step (+): ir a S2 en mode=create al final de la lista.
  • borrar step (d sobre el step apuntado, con confirmación): remover step + reindexar. Si rompe alguna referencia a previous_output, marcar el step afectado en rojo en el summary.
  • reordenar step (shift+↑ / shift+↓ sobre el step apuntado): mueve la posición del step en la lista. Re-evaluar dependencias previous_output.
  • abrir YAML en $EDITOR (y): suspende la TUI, abre $EDITOR sobre el archivo del pipeline (con bloque status incluido). Al cerrar el editor, re-parsea desde disco. Si el YAML quedó inválido → mostrar el error y mantener el modelo previo en RAM.
  • atrás (esc): ir al último step en mode=edit.
Validaciones pre-save
  • Pipeline tiene al menos 1 step.
  • Path destino libre (o usuario confirmó overwrite).
  • Cada step pasa el validador del modelo (ya validado en S2, doble check defensivo).

SC

Cancelled — confirm de salida

desde S1, S2 o S3
Trigger
esc en S1, o ctrl+c / q en cualquiera. Siempre dispara el modal — incluso con modelo vacío, para evitar salidas accidentales.
Render
Modal con 3 opciones:
  • keep & exit — el archivo queda guardado tal como está; aparecerá en My pipelines marcado como incompleto.
  • discard & exit — borra el archivo (si ya existía). Irreversible, con confirmación.
  • back — vuelve al estado anterior, modelo intacto.
Transiciones
  • "keep" → autosave sincrónico final → S0, descarta el modelo en RAM (vive en disco).
  • "discard" → os.Remove del archivo si existe → S0, modelo descartado.
  • "back" → vuelve al estado anterior.
bifurcaciones

Decisiones que cambian la UI dentro de un estado

B1 — Posición del step modifica el set de inputs disponibles

step idx == 0 Opciones input: text pr issue file url none. previous_output oculto.
step idx ≥ 1 Mismos 6 + previous_output. Si el usuario elige previous_output, los demás quedan disabled (mutex visual).

B2 — Validator on/off colapsa el bloque entero

validator: off Step se guarda con max_loops: 1 implícito y sin clave validator en el YAML. Los campos cli/kind/content/max_loops/on_max_loops del validator no se renderizan.
validator: on Aparecen los 5 campos. cli arranca con el mismo CLI que el step por default (más conservador, menos sorpresas). El usuario lo puede cambiar para habilitar cross-review. max_loops default 3, on_max_loops default fail.

B3 — Kind del step / validator alterna textarea ↔ skill picker

kind: prompt Render: textarea multiline (autosize). El YAML guarda el texto crudo en content.
kind: skill Render: SkillPicker filtrado por el CLI elegido. Lista ~/<cli-config>/skills/* + <cwd>/.<cli>/skills/* (alias para gemini → commands/*.toml). Si la lista viene vacía, mensaje "no hay skills instaladas para <cli>" + sugerencia de cambiar a kind: prompt o instalar una skill. El YAML guarda solo el nombre de la skill en content.

B4 — CLI no instalado al elegir cli del step

cli installed Pill habilitada normal.
cli no instalado Pill disabled con tooltip "no instalado · ver doctor". El pipeline asume que todos los CLIs elegidos están instalados; no se permite guardar un step apuntando a un CLI ausente.
modelo

Modelo interno acumulativo

Una sola struct vive en el bubbletea program raíz. Cada estado lee/escribe sobre la misma instancia. Al salir por SC se descarta; al salir por S4 se serializa. 

// internal/wizard/model.go
type Model struct {
    Screen   Screen            // S1 | S2 | S3 | S4 | SC
    Pipeline Pipeline          // el draft entero

    // estado UI específico de S2
    StepEdit StepEditState     // idx, mode, focus, errores

    // estado UI de S3
    SummaryCursor int          // step apuntado

    // detección de entorno (lazy, cacheada)
    Skills []skills.CLI        // reusa internal/skills.Detect

    // persistencia incremental
    Path        string        // ~/.che/pipelines/<slug>.yaml; "" hasta que S1 setee name
    SaveDirty   bool          // hay cambios pendientes de flush
    saveTimer   *time.Timer   // debounce del autosave async
}

type Pipeline struct {
    Status      *Status          // nil cuando es ready; presente cuando es draft
    Name        string
    Description string
    Steps       []Step
}

type Status struct {
    Stage       string            // info | step | summary
    StepIdx     int               // solo si Stage=step
    StepMode    string            // solo si Stage=step — create | edit
    LastSavedAt time.Time
}

type Step struct {
    Name        string
    CLI         string            // claude|codex|gemini|opencode
    Kind        string            // prompt|skill
    Content     string
    Input       string            // text|pr|issue|file|url|none|previous_output
    Validator   *Validator        // nil si off
    MaxLoops    int               // 1 si validator nil
    OnMaxLoops  string            // fail|continue|pause; "" si validator nil
}

type Validator struct {
    CLI     string
    Kind    string
    Content string
}

El YAML resultante (formato canónico, ya cerrado en design.html) se obtiene serializando Pipeline con gopkg.in/yaml.v3 en modo estricto. Campos nil / "" / 0 se omiten para mantener el archivo limpio.

roadmap

Plan de desarrollo — historias de usuario

Diez historias incrementales. Cada una entrega valor visible al usuario final, deja el wizard funcional al cierre, y se valida con un test e2e automatizado y un smoke manual reproducible. El orden es de dependencia: cada historia se construye sobre la anterior.

Cancel modal (Q4 keep/discard/back) entra transversal en H2 — la primera historia donde el wizard ya tiene algo que persistir.

che-funnel embebido como pipeline default

A partir de PR #98 (commit 7dc265c), el binario shipea un pipeline default llamado che-funnel embebido en internal/wizard/embedded/che-funnel.yaml via //go:embed y registrado en wizard.Builtins(). My pipelines ya no arranca vacío en una instalación nueva — la primera vez que el usuario abre el lister ve el row che-funnel con chip [default] aunque ~/.che/pipelines/ no exista o esté vacío.

El builtin es read-only por default: al ejecutarlo desde el lister arranca con el YAML embebido sin tocar disco. Si el usuario quiere personalizarlo, el lister soporta copy-on-edit — al activar e o y sobre el row builtin, el archivo se materializa en ~/.che/pipelines/che-funnel.yaml y a partir de ahí el override de disco gana sobre el builtin (mismo mecanismo que cualquier pipeline regular).

che-funnel vs v0.0.82 funnel

El pipeline replica el embudo clásico (idea → explore/plan → execute → close) con 4 steps secuenciales sobre claude, delegando la state machine de GitHub (labels che:*, issues, PRs, comments, worktrees) a los prompts vía tool use de gh y git. Diferencias intencionales con la versión vieja:

  • iterate desaparece como step manual — cada step tiene un validator con max_loops=3; si el verdict es changes_requested el step se re-corre con feedback hasta converger o agotar el cap (handled por R3 del runner nuevo, no por un step propio).
  • close postea un template de scoring (1-10 en completitud / fidelidad / alineación) como comment del PR. El humano lo llena fuera del pipeline — no hay gate humano mid-run.
  • Validator cross-CLI — cada validator usa codex en lugar de claude para reducir groupthink. Si el usuario solo tiene claude instalado, puede editar el YAML (copy-on-edit) o borrar el bloque validator de cada step.

Impacto en el lister

Las acciones existentes funcionan igual sobre el row builtin con dos diferencias:

  • d sobre el builtin no abre el modal de confirm: muestra un toast "default embebido — no se puede borrar" y sigue. Para "ocultar" el default, el usuario tiene que crear un shadow en ~/.che/pipelines/che-funnel.yaml (via copy-on-edit) y decidir qué hacer con él.
  • e y y sobre el builtin materializan el YAML en ~/.che/pipelines/che-funnel.yaml antes de abrir el editor (copy-on-edit). A partir de ahí el override del FS gana sobre el builtin — el merge en loadListItems dedupea por slug.
