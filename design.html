Claude CLI — flujos a cerrar

Un CLI en Go que estandariza el ciclo de trabajo con IA sobre issues de GitHub, para poder medir en mismas condiciones y detectar qué funciona y qué no.

stack: go + gh + CLIs de agentes github como backbone sin API keys: subprocess a claude/codex/gemini stateless: github como único estado cada agente comenta en github (issue o PR) comments con header iter/agent/role para tracking complejidad la asigna la IA veredicto final 100% humano
con IA = hace al menos un call a modelo sin IA = pura lógica local (gh, sqlite, input)
Flujos del CLI · 3 con IA · 2 sin IA
01

x idea

Anotar una idea como issue en GitHub. Claude Sonnet explora el codebase con tools (Read/Grep/Glob), decide si la idea es una sola unidad o si en realidad son varias ideas independientes y autocontenidas, y clasifica por tipo (feature / fix / mejora / ux) y size (XS/S/M/L/XL) cada una. Si splittea, crea N issues separados e independientes — sin jerarquía padre/hijo, sin referencias cruzadas. Un solo call estructurado con tools, baja temperatura. El CLI requiere estar parado en un repo con remote de GitHub; si no, aborta.

Diagrama de flujo
CLI
Pre-checks: git remote de GitHub · gh auth status · CLI claude disponible en el PATH si alguno falla → aborta. Los CLIs de codex/gemini se chequean cuando los seleccionás en explore/execute
↓
CLI
Menu principal 1. idea  ·  2. explore  ·  3. execute  ·  4. close  ·  5. eliminar
↓
Usuario
Elige 1 → idea · escribe el texto de la idea input libre, multilínea, tipo git commit
↓

↓
Claude Sonnet
Call estructurado único: decide si es 1 idea o N independientes, clasifica tipo y size de cada una, completa template(s) output JSON: { items: [{ type, size, title, body, justificación }, ...] } · si items.length === 1 no hay split, si ≥ 2 son ideas autocontenidas
↓
gh
Crea N issues, uno por cada item del output
· Todos los issues usan el mismo template completo (idea, contexto, criterios, notas, clasificación)
· Cada uno con sus labels: type:*, size:*, status:idea
· Si son N > 1, los issues son hermanos independientes sin referencias cruzadas
↓
CLI
Devuelve URL(s) de los issues creados y vuelve al menú
Template del issue (el mismo para todos los casos) ## Idea {idea parafraseada con claridad accionable, autocontenida} ## Contexto detectado - Archivos/módulos relevantes: {paths que Claude identificó} - Área afectada: {feature o dominio} - Dependencias visibles: {libs, otros módulos, APIs externas} ## Criterios de éxito iniciales - [ ] {criterio 1} - [ ] {criterio 2} ## Notas / warnings {complicaciones, edge cases o dudas que Claude detectó} ## Clasificación - Type: feature  ·  uno de: feature · fix · mejora · ux - Size: M — {justificación corta} Si Claude splittea la entrada en N ideas, se crean N issues con este mismo template, cada uno autocontenido. No hay links entre ellos.
Decisiones
  • Modelo: Claude Sonnet (4.6 o el más reciente)
  • Criterios de size: Claude lee el codebase con tools y estima por archivos tocados, lógica nueva, dependencias e impacto
  • Clasificación de tipo: feature · fix · mejora · ux — Claude la infiere del texto + contexto del codebase en el mismo call que el size
  • Decisión de split integrada: Claude decide en el mismo call si la entrada es una idea o son N ideas independientes y autocontenidas. Si son varias, crea N issues sin jerarquía ni referencias cruzadas. No hay comando split separado
  • Template único: Idea · Contexto · Criterios de éxito · Notas · Clasificación. Se usa el mismo para todos los issues creados, sin variantes para padre/hijo
  • Criterio de split: solo splittea si las ideas son verdaderamente independientes (pueden trabajarse por separado, tener sus propios PRs, cerrarse sin depender unas de otras). Si hay dependencia, es una sola idea
  • Repo destino: autodetect del remote en el pwd. Sin GitHub → el CLI aborta

02

x explore

Convertir un issue en estado status:idea en un plan ejecutable. El usuario elige un agente ejecutor (Opus / Codex / Gemini) que genera el plan, y entre 2 y 3 validadores (pueden repetir tipo, incluso los 3 iguales) que lo revisan en paralelo. Al terminar, el CLI le muestra al usuario el reporte completo y le ofrece 3 opciones: mandarlo, corregir y re-validar, o solo corregir. El plan final queda persistido en el issue y el label pasa a status:plan.

Template del plan (propuesto) ## Plan Generado por: {ejecutor} · Validado por: {validador1, validador2, ...} ## Steps 1. {step 1 — qué hacer + archivos a tocar} 2. {step 2} 3. ... ## Criterios de éxito (refinados) - [ ] {criterio 1} - [ ] {criterio 2} ## Tests - {tests a agregar o modificar} ## Riesgos / consideraciones - {riesgo + mitigación propuesta}
Formato de comments (validadores y correcciones) <!-- claude-cli: flow=explore iter=1 agent=codex instance=1 role=validator --> ## [validator:codex#1 · iter:1] Issues encontrados: - [A] {issue detectado, accionable} - [B] {otro issue} Correcciones del ejecutor (iter=2): <!-- claude-cli: flow=explore iter=2 agent=opus role=executor --> ## [executor:opus · iter:2 · correcciones] Atiende: - [A] {cómo se resolvió} - [B] {cómo se resolvió} Verificación scope-lock (iter=2): <!-- claude-cli: flow=explore iter=2 agent=codex instance=1 role=validator --> ## [validator:codex#1 · iter:2 · verificación] Checklist original: - [A] ✓ resuelto - [B] ✓ resuelto
Decisiones
  • Fuente de issues: se listan los que tengan status:idea en el repo del pwd
  • Ejecutores disponibles: Opus · Codex · Gemini (el usuario elige 1)
  • Validadores: entre 2 y 3, pueden repetir tipo (ej. 3 Opus, o 2 Codex). Corren en paralelo con goroutines
  • Ejecutor y validador pueden ser del mismo tipo: sí — ej. Opus ejecuta y Opus valida
  • Validador que falla: se ignora, el flujo sigue con los que respondieron (no aborta)
  • Conflictos entre validadores: se reportan explícitamente al usuario, no hay resolución automática
  • Iteración del plan: el usuario no se entera del nombre "iterate" — post-validación el CLI le ofrece 3 opciones inline (mandar / corregir y re-validar / sólo corregir)
  • Comments con header estructurado: cada comment de agente arranca con <!-- claude-cli: flow=explore iter=N agent=X role=executor|validator --> para que el CLI pueda parsear y pasar a la iteración siguiente solo lo relevante
  • Estado implícito por iter: el iter más alto de role=executor marca sincronización · validadores de iters anteriores se consideran addressed, los del último open · comments sin header se ignoran (pueden ser humanos)
  • Persistencia: plan en el body del issue (lo edita), comentarios de validadores como comments
  • Cambio de estado: status:idea → status:plan al terminar

03

x execute

Ejecutar un issue en estado status:plan. Trabaja siempre en un git worktree aislado y abre un PR draft. Los validadores leen el diff del PR (no el plan) y comentan en el PR. Al final se ofrecen las mismas 3 opciones que en explore. Incluye un mecanismo anti-loop-infinito para que las correcciones no generen nuevos bugs en cadena.

↓

1 Mandalo como está — continúa al cierre
2 Corregilo y volvé a validar — ejecutor aplica patch mínimo sobre el checklist fijado · validadores solo verifican ese checklist (no buscan nuevos bugs) · loop al reporte
3 Sólo corregilo y mandalo — ejecutor aplica patch mínimo y continúa al cierre sin re-validar
↓

PR queda listo (sigue draft o pasa a ready según config) · label: status:executing → status:ready-to-close
↓
CLI
Resumen: URL del PR + URL del issue + iteraciones usadas · vuelve al menú
Anti-loop-infinito: 3 mecanismos combinados 1. Scope lock Los validadores reportan problemas solo en la primera pasada. Ese output se congela como checklist fijo. En iteraciones siguientes los validadores reciben el prompt: "estos son los issues originales que vos reportaste. verificá uno por uno si se resolvió o no. NO busques problemas nuevos ni refactors sugeridos." 2. Patch mínimo en el ejecutor En cada iteración, el ejecutor recibe: "aplicá SOLO estos fixes del checklist. no refactorees código no relacionado. no agregues features nuevos. si un fix requiere cambios colaterales, documentá por qué." 3. Hard cap de iteraciones Máximo 3 iteraciones por run de execute. Al llegar a la 3ra, se manda igual: el PR queda listo para cerrar con los issues que no alcanzaron a resolverse. Si algo quedó pendiente, el veredicto humano en close lo captura.
Decisiones
  • Fuente de issues: se listan los que tengan status:plan en el repo actual
  • Ejecutor + validadores: misma selección que en explore (Opus/Codex/Gemini, 1 ejecutor + 2-3 validadores, pueden repetir tipo)
  • Worktree obligatorio: siempre en .worktrees/issue-{id}, branch exec/{id}-{slug}. No se toca el branch del pwd
  • PR draft automático: se abre al primer push, linkeado al issue con Closes #{id}
  • CI/CD como gate: el ejecutor no corre tests local. Después de pushear espera al CI de GitHub; si falla, corrige y re-pushea hasta que quede verde. Sin CI verde no se pasa a validadores
  • Validadores leen diff: no el plan completo. El contexto es el diff del PR + codebase del worktree
  • Comentarios en el PR: no en el issue — mantiene separado "qué discutimos del plan" vs "qué comentamos del código"
  • Comments con header estructurado: mismo formato que explore (<!-- claude-cli: flow=execute iter=N agent=X role=... -->). Se usa para que Opus sepa qué atender en iter+1 y los validadores hagan scope-lock sobre el checklist
  • Resolved conversations nativas: cuando Opus atiende un PR review comment, el CLI lo marca como resolved via API de GitHub. Dos mecanismos en paralelo — header para el CLI, resolved para que vos lo veas claro en la UI del PR
  • Post-validación: mismas 3 ramas que explore (mandar / corregir y re-validar / sólo corregir)
  • Anti-loop-infinito: scope lock del checklist + patch mínimo + hard cap 3 iteraciones. Al llegar a la 3ra se manda igual
  • Worktree al cerrar: se limpia durante el flujo close, no antes
  • Cambio de estado: status:plan → status:executing al iniciar, → status:ready-to-close al terminar

04

x close

Cerrar una idea ejecutada. El CLI lista los issues con status:ready-to-close, el usuario elige uno, puntúa del 1 al 10 en cada eje (completitud, fidelidad, alineación), escribe una nota libre. El CLI mergea el PR (lo que cierra el issue automáticamente por el Closes #{id}) y limpia el worktree. No interviene ningún modelo.

Decisiones
  • Fuente de issues: se listan los que tengan status:ready-to-close
  • Rúbrica: 3 ejes (completitud · fidelidad · alineación), 1 a 10 cada uno
  • Nota libre: sí, opcional, al final
  • Persistencia: comment final en el issue con scores + nota + metadata — GitHub como única fuente de verdad, sin SQLite
  • Merge del PR: el CLI lo mergea (gh pr merge --squash), lo que cierra el issue vía Closes #{id}
  • Conflicts en el merge: el CLI corta y avisa; el usuario resuelve a mano y reintenta close
  • Worktree: se limpia después del merge con git worktree remove
  • Label final: status:ready-to-close → status:closed

05

x eliminar

Descartar una idea en cualquier estado. Cierra el issue sin scoring, cierra el PR si hay (sin merge), borra el branch del remote si hay, y limpia el worktree si existe. Pide confirmación una vez. No interviene ningún modelo.

· Si hay PR abierto → gh pr close sin merge
· Si hay branch remote → git push origin --delete
· Si hay worktree → git worktree remove

Resumen de lo borrado · vuelve al menú
Decisiones
  • Alcance: cualquier issue en estado no-cerrado. Incluye los que nunca pasaron de status:idea
  • Confirmación: una sola vez con y/N. No hay flag --force porque el comando ya es destructivo por definición
  • Cleanup idempotente: si algo ya no existe (ej. worktree ya borrado), se skippea esa subacción sin error
  • Label final: status:deleted para diferenciar de status:closed (que pasó por el flujo completo y tiene scoring)
  • Sin scoring: no se pide rúbrica — el issue se descarta sin evaluarlo

Observabilidad

Cada flow corre varios subprocess a CLIs externos que pueden tardar minutos. Sin visibilidad en vivo el usuario cree que el CLI se colgó, y sin logs estructurados no se puede medir ni debuggear. Tres capas obligatorias.

01

Feedback en vivo en terminal

Cada subprocess emite líneas con timestamp + agente + estado. Permite ver qué está haciendo cada agente en tiempo real y detectar cuelgues o rate-limits sin abrir logs. Los validadores paralelos se muestran intercalados pero identificados por nombre y sufijo numérico.

[14:23:01 opus ] leyendo codebase (3 archivos) [14:23:15 opus ] generando plan... [14:23:42 codex#1 ] ✓ OK (2.1s) [14:23:48 gemini ] ✓ OK (2.7s) [14:24:01 codex#2 ] ⚠ rate limit · skippeado

02

Logs estructurados por run (JSONL) + blobs de prompts y outputs

Por cada call se guardan tres cosas: una línea en el JSONL del flow (metadata chico y grep-eable), el prompt completo en un blob, y el output completo del agente en otro blob. Los blobs se indexan por hash del contenido, así que si dos calls tuvieron el mismo prompt/output se dedupea automático. Todo vive en .claude-cli/logs/, gitignoreado. Con esto podés reconstruir cualquier call a posteriori para analizar si el agente alucinó, missió completitud, o se desvió — cruzás el output con los scores humanos de close y ahí está tu dataset de miss/hit.

# JSONL metadata (.claude-cli/logs/42/explore-2026-04-18T14-23-00.jsonl) {"ts":"2026-04-18T14:23:42Z","issue":42,"flow":"explore", "agent":"codex","instance":1,"role":"validator", "prompt_hash":"a1f2...","output_hash":"b3e4...", "tokens_in":1200,"tokens_out":450, "duration_ms":23100,"exit_code":0}
# Estructura de carpetas .claude-cli/logs/ ├── 42/ # issue 42 │ ├── idea-2026-04-18T10-12-00.jsonl # metadata │ ├── explore-2026-04-18T14-23-00.jsonl │ ├── execute-2026-04-18T15-01-00.jsonl │ └── blobs/ │ ├── a1f2...prompt.txt # prompt completo │ ├── b3e4...output.txt # output completo del agente │ └── ... └── 43/ └── ...
