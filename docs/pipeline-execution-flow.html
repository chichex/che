Flujo de ejecución de pipelines

Diagrama de estados, transiciones, bifurcaciones y validaciones de todo lo que es ejecución de pipelines: recolectar el input del primer step, preflight de dependencias, correr cada step en subprocess con streaming, loops del validator, persistencia de artefactos por run, cancel, finalización y vuelta al menu. Este doc refleja el estado implementado al cierre de H1..H10 + las addenda post-H10 (breadcrumbs, async loaders, prompt review en S2, "Crear pipeline con IA" en el menu, permission-denied chip, preflight de repo y otros polish que se sumaron despues del plan original). No cubre la creación ni edición del YAML — eso vive en manage-pipelines-flow.html.

Compañero de manage-pipelines-flow.html (que cubre el wizard S1–S4 y el lister de drafts/ready) — éste cubre lo que pasa cuando un pipeline ready se dispara desde My pipelines: enter sobre el row → este flow.

Punto de entrada único: My pipelines → enter sobre un row con chip ready. No hay invocación por subcomando ni por argumento — el run vive dentro de la TUI principal de che (el binario sin args abre la TUI; el menu home es Item 1 "My pipelines", Item 2 "Create pipeline", Item 3 "Crear pipeline con IA", Item 4 "See skills", Item 0 "Exit").

Header común — breadcrumbs. Toda pantalla del runner (R0..RF + modales RC/RP) abre con un breadcrumb que muestra la jerarquía completa con separador ›: che › My pipelines › Run · <name> › Running. El último segmento se pinta en cyan (titleStyle dracula); el resto en gris dimmed. Cuando se abre un modal, el último segmento muta a Cancel? / Pause mientras el modal está visible — el modal se renderea encima de R3, el breadcrumb refleja "dónde estoy realmente mirando ahora". Implementación común en internal/tui/tui.go.breadcrumb(), replicada en internal/runner/runner.go.breadcrumb() y en el wizard para que cada paquete pueda evitar imports cruzados.

artefactos

Persistencia del run

Cada run produce un directorio dedicado bajo ~/.che/runs/<slug>/<run-id>/. La idea es que un crash, un kill abrupto o un terminal cerrado dejen suficiente material en disco como para auditar qué pasó sin depender de que la TUI siga viva: stdout/stderr se flushean en cada línea, el manifest se reescribe atómicamente en cada transición de step, y el resultado del step queda parseado en un archivo aparte para que el step siguiente lo lea sin tener que re-tokenizar el stdout.

Un run = un directorio inmutable. Una vez terminado (done / failed / cancelled), nada lo edita. Re-correr el pipeline genera un run-id nuevo. Esto convierte ~/.che/runs/<slug>/ en un historial natural ordenado por timestamp.

Layout en disco

~/.che/runs/triage-checkout-flow/
  2026-05-08T14-32-11/                     # run-id = ISO sin colons
    manifest.yaml                          # estado del run + input_kind/input_value (atómico)
    step-01.stdout.log                     # raw stream del subprocess
    step-01.stderr.log
    step-01.events.jsonl                   # eventos parseados (solo si CLI=claude)
    step-01.result.yaml                     # resultado consumible por step N+1
    step-01.validator.01.stdout.log         # validator loop iter 1
    step-01.validator.01.stderr.log
    step-01.validator.01.verdict.yaml        # {verdict: ok|fail, feedback, raw_stdout truncado a 4 KiB}
    step-02.stdout.log
    ...
  2026-05-08T16-44-02/                     # otro run del mismo pipeline
    ...

Sin input.yaml separado. El plan original proponía un archivo aparte para el input recolectado en R1; al implementar caímos en cuenta de que input_kind + input_value son metadatos suficientes para auditoría y caben en el manifest sin agregar otro archivo. El resolved_payload (contenido del file/issue/url) NO se persiste — vive solo en memoria como RunModel.Input.ResolvedPayload y se consume al spawnear el step 0 (stdin del subprocess). Decisión pragmática: para casos típicos el payload ya está en otro lado (el issue, el archivo) — no vale duplicarlo.

events.jsonl solo para claude. spawn.go abre el archivo solo cuando step.CLI == "claude" (los demás CLIs corren en text mode sin stream-json — el parser cae a raw line-by-line y no hay eventos tipados que extraer). Para codex/gemini/opencode el archivo simplemente no aparece en el run dir.

Schema del manifest.yaml

Reescrito atómicamente (.tmp + os.Rename) en cada transición: start del run, start/end de cada step, start/end de cada validator loop, y cierre final (done / failed / cancelled / interrupted). La estructura es plana para que un cat rápido te diga el estado actual sin parsearlo entero. 

run_id: 2026-05-08T14-32-11
pipeline: triage-checkout-flow           # nombre del pipeline (Pipeline.Name del YAML)
pipeline_path: ~/.che/pipelines/triage-checkout-flow.yaml
started_at: 2026-05-08T14:32:11Z          # UTC, fijado al entrar a R3
finished_at: 2026-05-08T14:38:42Z         # vacio mientras status=running
status: done                              # running | done | failed | cancelled | interrupted
input_kind: issue                          # text|pr|issue|file|url|none — del step 0
input_value: "chichex/che#42"              # raw value tipeado/seleccionado en R1
steps:
  - idx: 1                                  # 1-indexed (alinea con step-01.* en disco)
    name: collect-signals
    cli: claude
    kind: prompt                          # prompt | skill
    status: done                          # pending | running | done | failed | cancelled
    exit_code: 0
    started_at: 2026-05-08T14:32:11Z
    finished_at: 2026-05-08T14:34:03Z
  - idx: 2
    name: cluster-anomalies
    cli: claude
    kind: prompt
    status: done
    exit_code: 0
    started_at: 2026-05-08T14:34:04Z
    finished_at: 2026-05-08T14:36:21Z
    validator:                              # bloque presente solo si el step lo tiene
      cli: gemini
      loops_run: 2
      max_loops: 3
      on_max_loops: fail
      final_verdict: ok                  # ok | fail | fail-but-continued | human-override
      last_feedback: ""                  # feedback del ultimo verdict.fail (si aplica)

Diferencias con el plan original: no hay pipeline_slug, current_step, total_steps ni che_version — el slug se infiere del path del run dir (~/.che/runs/<slug>/<run-id>/), el step actual sale de iterar steps[].status==running, el total es len(steps), y la versión la dimos por implícita (el binario que lo lee es el mismo que lo escribió). Sí hay input_kind + input_value — reemplazan al input.yaml separado del plan. final_verdict ahora distingue fail-but-continued y human-override (decisiones del modal RP) además del original ok|fail.

Schema del step-NN.result.yaml

El "output canónico" del step. Lo escribe el runner al cerrar el subprocess (siempre — incluso en exit ≠ 0, output queda con lo que haya en stdout y exit_code con el código real). El step N+1 con input: previous_output lee este archivo, no el stdout crudo — el campo output es el stdout completo del subprocess (sin parsing extra para mantener el contrato simple). 

step_idx: 1
step_name: collect-signals
exit_code: 0
output: |
  47 anomalías encontradas en logs de checkout entre 14:00 y 14:32...
  Cluster A (latencia P99 > 800ms): ...
  Cluster B (timeouts gateway): ...

Sin kind, content, duration_ms ni verdict. Esos campos del plan original quedaron fuera: kind y content ya están en el manifest (no agregan nada consumible al step siguiente); duration_ms se calcula del manifest (finished_at - started_at); el verdict vive en archivos dedicados (step-NN.validator.0K.verdict.yaml) — duplicarlo en el result generaría un punto extra de drift.

Schema del step-NN.validator.0K.verdict.yaml

step_idx: 1
loop: 2
verdict: ok                                # ok | fail (parser tolerante normaliza)
feedback: ""                              # cuerpo del feedback si verdict=fail
raw_stdout: |                                # stdout entero del validator, truncado a 4 KiB
  Sos un validador. ...
  verdict: ok

Cuándo se escribe cada cosa

EventoArchivos tocados
Entrar a R3 (start del run) Generar RunID, ejecutar GC del slug-dir (cap default 10, CHE_RUN_HISTORY), os.MkdirAll(runDir, 0700), escribir manifest.yaml con status: running, steps[].status: pending, input_kind + input_value seteados.
Step N arranca Update steps[N-1].status: running, started_at. Manifest atómico.
Línea de stdout/stderr del step Append a step-NN.stdout.log / .stderr.log. Sin atómico, sin batching: file.WriteString(line) + Sync() por línea.
Evento parseado del stream (solo claude) Append a step-NN.events.jsonl. Sirve para reconstruir la UI, auditar tool-use sin volver a parsear stdout, y para el post-mortem de permission_denials que renderea el chip warn en R4/RF (ver "Permission denied chip" más abajo). Para CLIs no-claude, el archivo no se abre.
Step N termina (exit) Escribir step-NN.result.yaml; update manifest steps[N-1].status: done|failed, exit_code, finished_at.
Validator loop K arranca Update manifest validator.loops_run: K; subprocess separado; eventos a step-NN.validator.0K.*.
Validator loop K cierra Escribir step-NN.validator.0K.verdict.yaml; update manifest. Si verdict: ok o K == max_loops, salir del loop.
Run termina (cualquier salida) Update manifest status + finished_at. Cierre de file handles. Trigger de GC.

Garantías y no-garantías

  • Atomicidad del manifest: .tmp + rename. Un crash entre dos transiciones deja un manifest viejo pero coherente, nunca uno trunco. Fault injection probado vía CHE_FAULT_INJECT_BEFORE_RENAME=1 (escribe el .tmp y aborta antes del rename — el manifest viejo queda intacto, el .tmp queda huérfano y se limpia en boot futuro).
  • Logs no son atómicos. file.WriteString(line) + Sync() por línea reduce el window a casi cero pero un kill -9 agresivo igual puede perder bytes en flight. Asumido — el manifest es la fuente de verdad.
  • Recovery de runs interrumpidos. Un manifest con status: running Y started_at > 1h al boot del lister se reescribe a interrupted (RecoverInterruptedRuns en manifest.go). La barrera de 1h es defensiva — manifests más nuevos pueden ser otro proceso de che corriendo en paralelo (no queremos pisarle el manifest mid-run).
  • GC: al iniciar un run nuevo, mantener los últimos N dirs por pipeline (mtime desc); más viejos → os.RemoveAll. Default N=10, override por env CHE_RUN_HISTORY=N; N=0 desactiva el GC.
  • Concurrencia: dos runs simultáneos del mismo pipeline son posibles porque el run-id los separa físicamente. No hay lock — el segundo no espera al primero. Out of scope v1.
lifecycle

Ciclo de vida end-to-end

Lo que pasa entre el enter sobre un row ready y la vuelta al menu, mirado como una secuencia de bloques en vez de un grafo. Los transitions ya están en el state diagram; esta sección documenta qué hace el código en cada bloque.

FaseTriggerOutput
1 · Resolución enter sobre row ready en My pipelines. Re-parsear ~/.che/pipelines/<slug>.yaml (no usar la copia en RAM del lister — puede estar stale si el usuario abrió $EDITOR). Validar IsValid() de nuevo. Si falla → toast en el lister, no se entra a R1.
2 · Input (R1) Pipeline válido + step 1 con input != none. Pantalla R1 que pide el input según el kind (textarea, picker de PR/issue via gh, file picker via TUI, URL input). Si input: none, R1 se skipea y va directo a R2.
3 · Resolve Usuario confirma input. Resolver el payload: gh issue view <ref> --json title,body, os.ReadFile, http.Get. Falla de fetch → R1 con error inline. Éxito → guardar el {Kind, Value, ResolvedPayload} en RunModel.Input in-memory (todavía no hay run-id ni archivo en disco — el input.yaml se persiste recién al entrar a R3, junto con el manifest).
4 · Preflight (R2) Input resuelto. Recorrer todos los steps y validar:
· CLI installed (reusar internal/skills.Detect).
· Si kind: skill, la skill existe en ese CLI.
· git repo context (solo si algún step usa input: pr|issue) — corre repoctx.Detect() y appendea el repo detectado al label como git repo context: owner/repo. Si InGitHubRepo == false, fail bloquea avance.
· gh auth status (solo si el pipeline necesita gh).
· File readable defensivo si input: file (re-check post R1).
· Disk space ≥ 100 MB libres en ~/.che/runs (warning amarillo, no bloquea).
R2 muestra checklist (✓/✗/!/⏳ por chequeo). El footer cambia según verdict — en all-OK muestra enter siguiente · esc volver (no aparece r reintentar porque no hay nada que reintentar); en has-fail aparece enter bloqueado · r reintentar · d abrir doctor · esc volver; en only-warn el primer enter pone preflightConfirm, el segundo avanza.
5 · Init run Preflight OK. Generar run-id (time.Now().UTC().Format("2006-01-02T15-04-05")); crear ~/.che/runs/<slug>/<run-id>/; escribir input.yaml + manifest.yaml inicial; trigger GC (mantener últimos N).
6 · Step loop (R3) Manifest creado. Por cada step del pipeline (en orden): resolver input (kind del step 1 ya está; para idx ≥ 1 con previous_output leer step-(N-1).result.yaml); spawn subprocess; tee stdout → .stdout.log + parser stream-json; tee stderr → .stderr.log; al exit, escribir result.yaml + actualizar manifest. Si tiene validator, entrar al validator loop.
7 · Validator loop Step terminó con exit_code: 0 + tiene validator. Spawn validator con step-NN.result.yaml/output como input. Parsear verdict: ok|fail + feedback al final del stdout. Si ok → siguiente step. Si fail y loops_run < max_loops → re-correr el step pasando el feedback como contexto extra. Si hit max_loops → aplicar on_max_loops.
8 · Terminal All steps done · o un step/validator falló · o el usuario hizo ctrl+c. Cerrar manifest con status: done|failed|cancelled, finished_at; transición a R4 (RunDone), RF (Failed) o RC (Cancelled). En todos los casos, el pipeline-yaml en ~/.che/pipelines/ queda intacto.

Detalle por estado

Cada estado describe trigger de entrada, render esperado, eventos del modelo, transiciones disponibles y qué tecla las dispara. Los estados R son screens; RP y RC son modales que viven sobre R3.

R1

InputPrompt — recolección del input del step 1

screen 1 / 4 del runner · internal/runner/input.go
Trigger
R0 emite "run pipeline X" + step[0].input != "none". Breadcrumb: che › My pipelines › Run · <name> › Input · <kind>.
Render dinámico
  • text · textarea multiline (cursor block ▎, shift+enter / alt+enter / ctrl+j para newline). Hint: "el step recibirá este texto en stdin / como prompt".
  • pr / issue · dos modos:
    • repo mode (default cuando repoctx.Detect().InGitHubRepo == true): picker async con los PRs / issues abiertos del repo activo. Mientras carga muestra Cargando PRs abiertos en owner/repo…. Al elegir un row, arma la referencia owner/repo#NNN y la pasa a resolveGH.
    • fallback libre (cuando no hay repo activo o el listado falla): single-line con la referencia tipeada a mano. R2 igual va a rebotar con "git repo context" fail — el textBuf existe para usuarios que sepan tipear refs de otros repos a mano.
  • file · file picker (dirs primero, luego files; ".." para subir; oculta dot-files). Tab no implementado v1 — alcanza con cursor + enter. Validación al confirmar: existe + no es dir.
  • url · single-line. Validación de schema (http/https). Fetch eager con timeout 10s al confirmar.
  • none · screen no aparece — enterFirstScreen salta directo a R2.
Carga async (pickers)
El picker de pr/issue (repo mode) inicializa con ghLoading=true y dispara loadGHListCmd desde RunModel.Init(). El primer frame muestra "Cargando…" en vez de bloquear el alt-screen mientras corre gh pr list. Mientras está cargando, ↑/↓/enter se ignoran en silencio (no hay lista que operar). Si gh falla, ghLoadErr se renderea como error inline y esc deja salir al lister.
Validaciones
  • Input no vacío (excepto none).
  • Resolución eager: si gh / http / os.ReadFile falla, error inline rojo bajo el input + foco vuelve al input.
  • Resolved payload no es vacío post-fetch (un PR sin body sigue siendo válido — solo bloqueamos si err != nil).
Transiciones
  • ctrl+s / enter sobre el último foco · validar + resolver + enterPreflight(m).
  • esc · vuelve a R0 (sin run iniciado, sin manifest creado).
  • ctrl+c · salida total de che.
Estado del modelo
Tras transición exitosa: RunModel.Input seteado con {Kind, Value, ResolvedPayload}. Sin escribir disco todavía — el run dir y el manifest se crean al entrar a R3.

R3

Running — full-screen takeover con steps + log streaming

screen 3 / 4 · loopable por validator · internal/runner/running.go
Trigger
R2 OK → enterRunning → initRunDir (GC + mkdir) → escribir manifest inicial → startStep(0).
Layout
che › My pipelines › Run · triage-checkout-flow › Running step 2/4 ✓ 1. collect-signals (claude) 2.3s ⏳ 2. cluster-anomalies (claude) · loop 1/3 · 3. review (claude) · 4. summarize (claude) [2/4] cluster-anomalies (claude / prompt) > Reading collect-signals.json... > Found 47 anomalies clustered into 6 groups · tool: read_file > Validator (gemini) loop 1/3... █ ctrl+c cancelar · ↑/↓ scroll · g auto follow · ctrl+l clear · tab cambiar logs
  • Breadcrumb: che › My pipelines › Run · <name> › Running. Cuando hay modal RC abierto el último segmento muta a Cancel?; con RP, a Pause.
  • Sub-header: step N/M en gris dimmed (debajo del breadcrumb, no embebido).
  • Steps tracker: un row por step del pipeline. Iconos: ✓ done (verde), ✗ failed (rojo), ! cancelled (amarillo), ⏳ running (amarillo bold), · pending (gris). El row activo tiene el label en labelStyle (blanco bold). Cuando el validator del step activo está corriendo, el row appendea · loop K/max.
  • Log pane: header del step en LogFocus con label "[N/M] name (cli / kind)"; si LogFocus != Active appendea "· viendo logs (tab para cycle)". Body = ring buffer del step (cap 2000 líneas) con sticky-bottom por default. Tamaño dinámico: si llegó tea.WindowSizeMsg, calcula terminalHeight - chrome (8) - rows del tracker; si todavía no, fallback a 18 líneas. Piso de 4 para terminales chiquitos.
  • Footer: ctrl+c cancelar · ↑/↓ scroll · g auto follow · ctrl+l clear + · tab cambiar logs cuando hay >1 step.
Render del log pane
  • claude: parser de stream-json --verbose (internal/runner/parser/claude.go). Buffer del scanner subido a 1 MiB (memory bufio.Scanner 64 KiB silent drop). Eventos crudos van a step-NN.events.jsonl, líneas humanas (> assistant text, · tool: read_file) van al ring buffer.
  • codex: exec --json; parser stub que cae a raw line-by-line (el shape estable está pendiente — cuando esté, el parser dedicado lo cubre sin tocar el spawn).
  • gemini / opencode: text mode, sin stream-json. Stdout crudo línea por línea via parser.Raw(). Sin events.jsonl en disco.
  • Stderr en rojo dimmed (stderrStyle), intercalado con stdout en orden de llegada.
  • Indicador "(scroll · N líneas más abajo · g para ir al fondo)" cuando StickyBottom=false y LogScrollOffset > 0.
Transiciones
  • ↑/↓ (también k/j) · scroll del log pane (desactiva sticky-bottom; al volver al fondo lo reactiva).
  • g · re-stick al fondo + reset scroll.
  • ctrl+l · limpia el ring buffer del step activo (los logs en disco quedan; solo afecta render).
  • tab · cyclea LogFocus entre 0..N-1 (solo aplica si hay >1 step). El log pane pasa a renderear los logs del step seleccionado sin frenar el subprocess en curso.
  • ctrl+c · abre RC modal (cancel confirm) — siempre, el run no es "salida accidental".
  • Auto (handleStepDone): step done sin validator → advanceAfterValidator → siguiente step (o R4). Step done con validator → startValidator (loop K=LoopsRun+1). Step exit_code != 0 o spawnErr → RF inmediato (stop-on-error: los siguientes nunca se inician y quedan en pending en el manifest).
  • Auto (handleValidatorDone): verdict ok → advance; verdict fail + loops < max → rerunStepWithFeedback (re-spawnea el step con el feedback prependeado al payload original); hit max + on_max_loops=fail → RF; hit max + continue → FinalVerdictFailButContinued + advance; hit max + pause → modal RP.
Modelo
RunModel.Steps[i] con Idx, Name, CLI, Kind, Status, StartedAt, FinishedAt, ExitCode, SpawnError, Validator *ValidatorRun, PermissionDenials []string; RunModel.LogBuffers[i] ring buffer de 2000 líneas por step (RAM); runState puntero compartido con la goroutine del spawn (cmd, requestCancel, lineCh, done, cancelled).
R4

terminal · internal/runner/done.go
Trigger

Cancelled — confirm de cancelación

Trigger

Render
Modal con borde rojo (errorStyle) superpuesto al log pane, con 3 opciones:
  • abort & save (default) — manda requestCancel al runState. La goroutine cancel hace SIGTERM al pgid; tras CHE_KILL_GRACE (default 5s) → SIGKILL. handleStepDone recibe el stepDoneMsg con Cancelled=true y ExitCode=-1 (forzado), cierra manifest con status: cancelled, transiciona a RF (con tono amarillo).
  • back to run — cierra el modal (CancelModal=false); el run sigue como estaba.
  • exit che — requestCancel + exitApp=true + tea.Quit. La TUI cierra inmediatamente; la goroutine de cleanup mata el subprocess en background.
Comportamiento (H9 polish)
  • El runner SIEMPRE intenta SIGTERM antes que SIGKILL — los CLIs pueden estar haciendo cleanup (cerrar archivos, terminar tool calls, llamar abort de la API).
  • Gracia configurable env CHE_KILL_GRACE (entero en segundos), default 5s.
  • Process group propio (setProcAttrs) para que el SIGTERM al cmd alcance a los hijos del subprocess (defensivo — los CLIs pueden spawnear helpers).
  • Sync() + Close() de stdout/stderr/events files antes del stepDoneMsg. Los logs en disco quedan consistentes y un re-open posterior no devuelve EBADF.
  • Mientras el modal está abierto, updateCancelModal sigue drenando stepLineMsg (los appendea al ring buffer y re-issue del wait) — sin esto el canal se llenaría y la goroutine de tee se bloquearía. stepDoneMsg y validatorDoneMsg que lleguen mientras el modal está visible se procesan igual (la transición a RF/R4 los recoge).
  • Race "subprocess salió por su cuenta entre ctrl+c y la decisión del modal": si cancelled=true pero el subprocess salió limpio (exit 0, sin SpawnErr), handleStepDone trata el step como done — el doc lo deja explícito: "tratar como step normal terminado".

addenda post-H10

Features que se sumaron después del plan H1..H10

Estas piezas no estaban en el plan original (H1..H10) pero quedaron en el binario final porque cubren modos de falla que aparecieron al usar el runner en condiciones reales. Cada una es ortogonal al state diagram principal — viven como decoraciones / pre-validaciones / UX que el plan no anticipaba.

1 · Async loaders ("Cargando…")

Problema: alt-screen + reads sincrónicos de filesystem o subprocess (gh) dejaban la pantalla en blanco varios cientos de ms al entrar a una screen — el usuario veía "todo desaparecer y reaparecer".

Fix: dos surfaces lo aplican.

  • R0 lister "My pipelines" (internal/wizard/list.go.loadListItemsCmd): el primer frame muestra Cargando pipelines… en gris dimmed mientras el goroutine lee ~/.che/pipelines/*.yaml + el último manifest por row. Al volver, listLoadedMsg repuebla la lista.
  • R1 picker pr/issue (repo mode) (internal/runner/input.go.loadGHListCmd): mientras corre gh pr list / gh issue list, el picker muestra Cargando PRs abiertos en owner/repo…. El handler ignora ↑/↓/enter en silencio mientras ghLoading=true; ghListLoadedMsg repuebla ghEntries al volver.

2 · Wizard S2 — review automática del prompt con claude -p

Implementación: internal/wizard/promptreview/ (wrapper sobre claude -p con system prompt + parser tolerante de JSON) + internal/wizard/step_review.go (bubbletea handler).

Cuando el usuario confirma un step (ctrl+s, ctrl+n, o "save & finish" del modal de save choice) con kind=prompt + content no vacío, shouldRunPromptReview dispara una review async que evalúa 4 criterios:

  1. Imperativo — pide acciones concretas, no análisis abierto.
  2. No interactivo — no asume AskUserQuestion ni confirmación humana mid-step (modo de falla típico en no-TTY: claude pide permiso, queda colgado, exit 0 sin trabajo real).
  3. Sin ambigüedad — estado final esperado bien definido.
  4. Tools claras — si necesita Bash/gh/git, mencionarlas; si no debe usar ciertas tools, prohibirlas.

El modal ScreenStepReview arranca en estado loading ("Analizando prompt con claude…", solo esc para cancelar) y, al volver, muestra Issues encontrados + Summary + Sugerencia (si hay) con opciones:

  • 1 guardar igual (ignorar review)
  • 2 aplicar sugerencia y guardar (solo si Suggested no vacío)
  • 3 volver y editar
  • r re-review tras editar

Si OK=true y sin issues, el modal se saltea automáticamente y persistimos el step (caso "review verde sin sorpresas"). El env CHE_DISABLE_PROMPT_REVIEW=1 apaga el feature entero — lo usa el harness e2e (claude está fakeado y no tiene matcher para la review; pedir uno por cada test sería ruido).

3 · Home menu — opción "Crear pipeline con IA"

Implementación: internal/tui/aigen.go + internal/aiprompt/ + internal/clipboard/.

Item 3 del menu home (tui.ActionAIGen). Pantalla con dos campos (Nombre + Descripción) → ctrl+s arma un prompt sanitizado y lo copia al portapapeles. El prompt encapsula:

  • Schema literal del YAML de pipeline / step / validator con todos los enums (cli, kind, input, on_max_loops).
  • Análisis de viabilidad obligatorio al inicio: verdict recomendado / overkill / no-recomendado / dudoso + razonamiento + recomendación (si no-recomendado, no genera YAML).
  • Reglas obligatorias: imperativo, no-interactivo, sin AskUserQuestion, tools claras, previous_output en steps ≥ 1, slug del name en kebab-case.
  • Ejemplo completo de un pipeline bien formado (triage-checkout-flow).

El copy al portapapeles usa el binario nativo de cada plataforma (internal/clipboard/clipboard.go): pbcopy en macOS, wl-copy > xclip > xsel en Linux, clip.exe en Windows. Si nada está disponible, devuelve ErrNoCopier y la pantalla muestra "no se pudo copiar — copialo manualmente abajo" como warn (el prompt sigue visible con scroll). y re-intenta la copia (útil si el usuario cambió de portapapeles entre apps).

4 · Permission denied chip (R4/RF)

Implementación: internal/runner/permdenied.go (post-mortem reader) + internal/runner/running.go (handler) + internal/runner/done.go (render).

Modo de falla cubierto: claude pide una tool en no-TTY (típicamente AskUserQuestion, pero también Bash interactivo), no hay TTY → la tool queda denegada → claude termina con exit 0 sin haber hecho nada útil. El verdict "done" del manifest se preserva (semántica correcta del subprocess), pero el usuario necesita una señal de que el step no hizo nada.

Al cerrar el step, handleStepDone llama a readPermissionDenials que abre step-NN.events.jsonl, busca el último evento type=result y extrae los permission_denials[].tool_name. Si el array no está vacío, se popula StepRun.PermissionDenials. En R4/RF:

  • Chip global ámbar al lado del breadcrumb: ⚠ permission denied — revisar prompt.
  • Por step row: appendea ⚠ AskUserQuestion, OtraTool en ámbar al lado del ✓.

Solo aplica a step.CLI == "claude" (los demás CLIs no emiten stream-json; el archivo no existe y la función devuelve nil sin error). Filtro barato: solo unmarshalea líneas que contienen "type":"result" como substring (los assistant/tool events son la mayoría; saltarlos por substring evita 99% de los json.Unmarshal).

5 · Pills pr / issue deshabilitadas en el wizard sin repo

Implementación: internal/wizard/model.go.inputDisabled + internal/wizard/step.go (render + cursor + buildStep).

Cuando el wizard renderea las pills de input en S2 y repoctx.Detect().InGitHubRepo == false, las pills pr y issue se renderean dim con sufijo ·off:

text ·off pr ·off issue file url none previous_output

El cursor de selección las salta automáticamente (stepNextFocus respeta inputDisabled); el shortcut directo (digit 2/3 = pr/issue) tampoco las setea. buildStep tiene un guard defensivo: si por alguna razón el state queda con input=pr sin repo, devuelve error en vez de persistir un step que va a rebotar en R2.

Combinado con el chip [needs repo] del lister + el preflight git repo context de R2, cubre la regla de "el usuario no debería poder elegir pr sin repo, y si igual lo elige (yaml editado a mano), el runner se lo dice con un fail bloqueante en preflight".

6 · Wizard polish

  • esc en S1 sin path en disco: cuando es una creación fresca (no avanzó a S2 todavía, no hay archivo persistido) sale directo al menu sin abrir el modal SaveChoice. El SC modal solo aparece cuando hay algo que perder.
  • Wrap dinámico de inputs de texto: los campos Nombre, Descripción, Prompt, Validator Prompt y Nombre del step se wrappean al ancho del terminal via tea.WindowSizeMsg. Sin esto, terminales angostos cortaban texto a la derecha.

7 · Runner — desviaciones del plan H1-H10

Detalles de implementación que difieren del plan pero son ortogonales al state diagram:

  • RC y RP como flags sobre R3 (no screens aparte). m.CancelModal bool y m.PauseModal *PauseState conviven con ScreenRunning; el breadcrumb muta el último segmento (Cancel? / Pause) para reflejar el sub-screen activo. Esto simplifica el dispatch del Update (todas las teclas pasan por updateRunning) y evita duplicar el render del log pane bajo el modal.
  • defaultSpawnCmd por CLI: claude recibe -p <content> --output-format stream-json --verbose. Codex recibe exec --json <content> (parser stub que cae a raw — el shape estable está pendiente). Gemini recibe -p <prompt> o /<skill> según kind (text mode). Opencode: run <prompt> (text mode).
  • events.jsonl solo para claude: spawn.go abre el archivo solo cuando step.CLI == "claude". Los demás CLIs lo dejan nil y no escriben — el dir queda más limpio y el post-mortem de permission denials se aplica solo donde tiene sentido.
  • Manifest atómico siempre (no solo en H8): writeManifest usa .tmp + os.Rename en TODAS las llamadas (start del run, start/end de cada step, start/end de cada validator loop, cierre). Fault injection testeable via CHE_FAULT_INJECT_BEFORE_RENAME=1.
  • GC con cap default 10, CHE_RUN_HISTORY=N para override. N=0 desactiva el GC.
  • Recovery al boot reescribe a interrupted los manifests con status: running + started_at > 1h (RecoverInterruptedRuns en manifest.go; lo invoca el lister antes de renderear).
  • SIGTERM → SIGKILL escalation: signalCancel manda SIGTERM al pgid, espera CHE_KILL_GRACE (default 5s, polling cada 50ms para detectar exit antes), después SIGKILL.
  • Retry desde RF con r: re-arma el program (state limpio, run-id nuevo) preservando solo el InputState. Implementación: retryRequested bool + loop en runner.Run que vuelve a invocar tea.NewProgram(m).Run() con el input pre-cargado.
  • Validator parser tolerante: parseVerdict parte el stdout en bloques (separados por blank line o ---), itera de atrás hacia adelante intentando yaml.Unmarshal; el primero que parse y tenga clave verdict no vacía gana. Cualquier valor distinto de "ok" se mapea a "fail" (verdict: failed / verdict: KO no engañan al loop). Sin bloque → fail + feedback: "no verdict block".
  • Validator se invoca con preamble + stepStdout via stdin (mismo patrón que el step original). El preamble pide explícitamente el bloque YAML al final.
bifurcaciones

Decisiones que cambian la UI o el comportamiento

B1 — Tipo de input del step 1 cambia R1

input: text Textarea multiline. Resolución: el texto crudo es el payload.
input: pr / issue Single-line input + helper que sugiere repos del $PWD (lee git remote -v + cache de gh repo list). Resolución: gh pr view --json title,body,comments. Falla → R1 con error inline.
input: file File picker minimalista con tab para nav + filtro inline. Resolución: os.ReadFile + check de tamaño.
input: url Single-line. Resolución: http.Get con timeout 10s, body como string. Status no-2xx → error inline.
input: none R1 se skipea. input.yaml tiene step_input_kind: none + resolved_payload: "".

B2 — Validator on/off cambia el flow del step

validator: nil Step termina → siguiente step. manifest.steps[i].validator ausente.
validator: presente Step termina con exit_code: 0 → spawn validator con el output del step como input. Parsear verdict. Loopear hasta ok o max_loops. Si fail final → aplicar on_max_loops.

B3 — Resolución de previous_output

step idx == 0 Imposible por construcción — IsValid ya bloqueó esto en el wizard. Defensive: si llegara a aparecer, RF inmediato con error "previous_output en step 0".
step idx ≥ 1 Leer step-(N-1).result.yaml/output y pasarlo como input del step N. Si el step previo terminó con validator activo, leer del último loop exitoso.

B4 — on_max_loops

fail Manifest final_verdict: fail · run termina en RF.
continue Manifest final_verdict: fail-but-continued · acepta el último output del step y pasa al siguiente. Útil para validators "mejor esfuerzo" donde la última iteración igual sirve.
pause Trigger de RP modal · espera decisión humana.

B5 — Render del log pane según CLI

claude / codex (stream-json --verbose) Parser de eventos: {type:assistant, content} → línea humana, {type:tool_use, name, input} → "· tool: name". Buffer 1 MiB del scanner para evitar el truncate silencioso de 64 KiB (ver memory).
gemini / opencode (text) Stdout crudo line by line. Sin parsing.

Fixes y polish post-2026-05-09

Lote chico de fixes sobre R1 y R3 que aparecieron al usar el runner en corridas reales del che-funnel embebido. Ninguno cambia el state diagram; son fixes de presentación y de robustez del streaming.

1 · R1 — wrap del prompt stdin al ancho de terminal

Commit 5696e5d. El textarea de R1 (kind=text) no envolvía: textos largos quedaban en una sola línea que se cortaba contra el borde derecho del alt-screen y el caret salía del viewport. Fix: aplicar wrap soft al width visible del runner.

2 · R3 — wrap del log pane streaming al ancho de terminal

Commit 443860a. El log pane viviente (sticky-bottom + ring buffer) recibía líneas crudas del subprocess; cuando el CLI shippeaba payloads largos sin \n intermedio, la línea tapaba el borde derecho y el footer. Fix: wrap por terminal width antes de imprimir.

3 · R3 — unstick del validator sobre stream-json

Commit 92a6f52. El validator del che-funnel (codex con stream-json --verbose) podía quedar trabado esperando un EOF que nunca llegaba si el último evento del stream no cerraba un bloque verdict: reconocible — el parser caía a "no verdict block" pero el reader del subprocess seguía abierto. Fix: liberar el reader explícitamente al detectar fin de stream sin verdict, dejando que parseVerdict emita el fallback fail con el feedback raw.

4 · R3 — stream del output del validator al log pane

Commit 03de057. Antes el output del validator solo aparecía en events.jsonl y en el feedback re-payload, no en la UI viviente — el usuario veía "validating…" hasta el verdict. Fix: el validator usa el mismo log pane que el step, con prefijo distintivo, así el usuario ve qué está haciendo el cross-CLI review en tiempo real.

5 · Bug chain del runner del che-funnel (PR #107)

Commit 04fe67b. Cinco fixes encadenados que aparecieron al correr el che-funnel embebido y son ortogonales al state diagram:

  • Fix 1 (raíz): interpolar {{INPUT}} en step.Content y step.Validator.Content antes de armar args. Helper interpolateInput invocado desde defaultSpawnCmd y runValidator. El payload sigue viajando por stdin (compat).
  • Fix 2: splitVerdictBlocks reconoce fences markdown (```yaml / ```yml / ``` sin info-string) como candidatos adicionales. Antes, verdicts envueltos en un fence dentro de prosa caían al fallback "no verdict block".
  • Fix 3: nuevo flag Verdict.RawFeedbackOnly + ValidatorRun.LastFeedbackRawOnly. Si el feedback es sólo el fallback raw "verdict: <token>", rerunStepWithFeedback suprime el feedback al payload (no re-prompt con ruido). Defensa adicional vía isRawVerdictFallback en mergeFeedbackIntoPayload.
  • Fix 4: rotación de events.jsonl por corrida. StepRun.EventsRun (1-based) se incrementa en startStep y rerunStepWithFeedback. El archivo se escribe como step-NN.events.RUN-K.jsonl — los reruns por validator ya no pisan el log original.

6 · R3 — feedback accionable cuando el validator sale con exit != 0 (PR #117)

Hasta este fix, cuando el subprocess del validator terminaba con exit_code != 0, runValidator seteaba Verdict.Feedback = "validator exit N" y descartaba el stdout/stderr capturados. El step reintentaba a ciegas con ese string pelado, derivando en cascada de role-confusion (el modelo del step empezaba a emitir verdict: approve en su propio output).

Fix: el nuevo helper buildInfraFailFeedback(cli, exitCode, stdout, stderr) en internal/runner/validator.go aplica el siguiente orden de fallback:

  1. Codex stream-json error event: si cli == "codex" y el stdout del validator contiene una línea JSON con {"type":"error","message":"..."} (o "type":"turn.failed"), se extrae el message. Feedback: "validator (cli=codex) infra-failed exit=N: <message>". Helper auxiliar: extractStreamJSONErrorMessage(stdout) (itera línea por línea, retorna "" si nada matchea).
  2. Tail de stderr/stdout: si no hay error event parseable (cualquier CLI), se toman las últimas 10 líneas no vacías de stderr; si stderr está vacío, de stdout. Feedback: "validator (cli=<X>) infra-failed exit=<N>: <últimas líneas>".
  3. Fallback compat: si ambos stdout y stderr están vacíos, retorna "validator exit N" (comportamiento anterior, sin marcar InfraFail).

Wrapping anti role-confusion: en los casos 1 y 2, el feedback se envuelve con el prefijo "El validator no pudo evaluar tu output por una falla de infra (no por un problema del trabajo). Reintenta el step manteniendo el output. Detalle: …". El wrapping vive dentro de buildInfraFailFeedback para que Verdict.Feedback persistido en verdict.yaml sea el string accionable final.

Cap de 4 KiB: el resultado se trunca reusando truncateForRecord (s[:4096] + "\n... (truncado)"), la misma función que limita el raw_stdout de verdict.yaml. El cap se aplica después del wrapping para acotar el contexto del rerun sin recortar el prefijo explicativo.

Campo Verdict.InfraFail bool: nuevo campo en la struct Verdict (validator.go) con tag yaml:"-" (no persistido en verdict.yaml). Se setea a true en los casos 1 y 2 de buildInfraFailFeedback. Paralelo a Verdict.RawFeedbackOnly; disponible para futuros consumidores (por ejemplo, una opción on_validator_infra_fail en el schema del step) sin cambiar el shape del manifest YAML.

Propagación de stderr: runValidator ahora propaga stderrBuf.String() al campo RawStderr del validatorDoneMsg (paralelo a RawStdout), necesario para que buildInfraFailFeedback pueda examinar stderr sin leer el archivo de log.

mergeFeedbackIntoPayload y isRawVerdictFallback (running.go) no se tocan: el feedback wrappeado nunca matchea el patrón ^verdict: <token>$, así que el filtro defensivo no lo filtra y el prepend funciona sin cambios.
